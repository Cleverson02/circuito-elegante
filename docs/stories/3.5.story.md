## Descricao

Este story implementa o **Mapeamento de Estado Oculto** — o mecanismo que permite ao hospede selecionar uma opcao de quarto dizendo "quero a segunda" ou "a mais cara", e Stella resolver automaticamente para o `offerId` exato sem jamais expor identificadores tecnicos (roomId, ratePlanId, offerId) na conversa.

**Problema/Necessidade:** Apos a curadoria (Story 3.4) apresentar 3-4 opcoes ao hospede, ele precisa indicar sua escolha. Num concierge humano de luxo, o hospede simplesmente aponta ou diz "aquela suíte com vista para o mar" — jamais informa um codigo. Stella deve replicar essa experiencia: o hospede responde com ordinal ("a segunda"), superlativo ("a mais barata"), posicao relativa ("a do meio"), ou faz Reply a uma imagem especifica no WhatsApp. Em todos os casos, Stella resolve internamente para o payload tecnico completo, mantendo a ilusao de conversa natural.

**Paradigma UX (PRD §3.3 — Correfencia Implicita):** "Hospede responde com pronomes ou posicao ('a segunda', 'a mais cara'); Stella resolve automaticamente." O hospede NUNCA ve requestId, offerId, roomId — estes sao tokens opacos que existem apenas no estado interno Redis.

**Decisao Arquitetural:** O mapeamento e implementado como um **OfferMap** armazenado em Redis no escopo da sessao. Cada opcao apresentada pela curadoria recebe uma posicao numerica (1-4) e, opcionalmente, um `whatsappMessageId` quando a renderizacao premium (Story 4.4) envia imagens. O `CoreferenceResolver` recebe o texto ou messageId do hospede e retorna o payload tecnico correspondente, ou solicita esclarecimento se a referencia for ambigua.

**Fluxo Completo:**
```
Curadoria (3.4) → OfferMap (Redis) → Hospede responde "a segunda"
                                         ↓
                                   OrdinalParser → posicao 2
                                         ↓
                                   OfferMap.options[1] → { offerId, roomId, ratePlanId, totalPrice }
                                         ↓
                                   Orchestrator recebe offerId → Quotation (3.3)
```

### OrdinalParser — Patterns por Idioma

**Estrategia de matching:**
1. Normalizar input: lowercase, trim, remover acentos (`normalize('NFD').replace(/[\u0300-\u036f]/g, '')`)
2. Tentar superlativo primeiro (patterns mais longos, maior especificidade)
3. Tentar posicao relativa
4. Tentar ordinal
5. Se nenhum match: `{ type: 'none' }`

**Se `language` nao fornecido:** Testar TODOS os 3 idiomas em sequencia (PT → EN → ES). Como max 4 posicoes e patterns nao colidem entre idiomas, nao ha ambiguidade.

### WhatsApp Message ID — Formato Evolution API

A Evolution API retorna o message ID no callback de envio de mensagem. O formato tipico e:

```
"3EB0XXXX..." (WhatsApp internal ID, ~32 chars hex)
```

Quando o hospede faz Reply a uma mensagem, o webhook da Evolution API inclui o campo `contextInfo.stanzaId` (ou `message.contextInfo.quotedStanzaId`) que contem o messageId da mensagem original. A Story 4.5 extraira este campo e passara para `resolveCoreference({ whatsappMessageId })`.

**Nesta story:** O resolver aceita `whatsappMessageId` como string opaca e faz lookup direto no OfferMap. A integracao real com o webhook da Evolution API e responsabilidade da Story 4.5.

### Integracao com Orchestrator

O Orchestrator Agent (pipeline principal) chama `resolveCoreference()` quando detecta que:
1. O contexto atual e "selecao de opcao" (hospede ja recebeu opcoes curadas)
2. A mensagem do hospede contem referencia posicional ou e um WhatsApp Reply

```typescript
// Pseudo-codigo — Orchestrator integration (fora do escopo desta story)
if (agentState === 'AWAITING_SELECTION' && offerMapExists(sessionId)) {
  const result = await resolveCoreference(sessionId, {
    text: guestMessage.text,
    whatsappMessageId: guestMessage.contextInfo?.quotedStanzaId,
    language: session.language,
  });
  
  if (result.resolved) {
    // Proceed to quotation with result.option.offerId
    await createQuotation({
      requestId: result.option.requestId,
      offerId: result.option.offerId,
      customerId,
    });
  } else if (result.clarificationNeeded) {
    // Ask guest to clarify
    return result.message;
  } else if (result.mapExpired) {
    // Suggest new search
    return result.message;
  }
}
```

### Security — Zero Technical ID Leakage

**Regra inegociavel:** Nenhum identificador tecnico (offerId, roomId, ratePlanId, requestId) deve aparecer em:
- Mensagens enviadas ao hospede (text, caption)
- Mensagens de clarificacao ("Qual opcao voce gostaria?" — nunca "Qual offerId?")
- Mensagens de erro ("Nao encontrei essa opcao" — nunca "offerId invalido")
- Logs de nivel `info` (somente `debug` pode logar IDs para troubleshooting)

**IDs so transitam em:**

### Resolucao de Superlativo — Algoritmo

```typescript
function resolveBySuperlative(
  offerMap: OfferMap,
  criteria: 'cheapest' | 'most_expensive'
): OfferOption {
  const sorted = [...offerMap.options].sort((a, b) => a.totalPrice - b.totalPrice);
  return criteria === 'cheapest' ? sorted[0] : sorted[sorted.length - 1];
}
```

**Edge cases:**
- Se 2+ opcoes tem o mesmo preco mais baixo/alto, retorna a de menor posicao (stable sort)
- Se apenas 1 opcao, retorna ela independente do criteria

### Resolucao de Posicao Relativa — Algoritmo

```typescript
function resolveByRelativePosition(
  offerMap: OfferMap,
  criteria: 'middle'
): OfferOption {
  const mid = Math.ceil(offerMap.options.length / 2);
  return offerMap.options[mid - 1]; // 1-indexed → 0-indexed
}
```

**Resultados por quantidade:**
- 1 opcao: `ceil(1/2)` = 1 → opcao 1
- 2 opcoes: `ceil(2/2)` = 1 → opcao 1
- 3 opcoes: `ceil(3/2)` = 2 → opcao 2
- 4 opcoes: `ceil(4/2)` = 2 → opcao 2
