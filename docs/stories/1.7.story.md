## Descrição

Criar pipeline de ingestão v2 que lê os 92 JSONs enriquecidos de `data/enrichment/*.json`, popula o JSONB `data` da tabela `hotels` seguindo a taxonomia de 58 campos, extrai campos booleanos para colunas schema, cria índices otimizados, e implementa a tool `query_hotel_details` para a Stella consultar dados estruturados dos hotéis.

**Problema/Necessidade:** Os 92 hotéis têm dados ricos coletados (83-95% completude, ~50 campos cada) mas o JSONB `data` no banco está vazio (`{source: 'xlsx-ingest'}`). Os campos booleanos `pet_friendly`, `pool_heated`, `bradesco_coupon` estão hardcoded como `false`. A Stella não consegue acessar nenhum dado enriquecido — amenidades, políticas, gastronomia, atividades são todos invisíveis.

**Visão:** Após esta story, a Stella terá acesso direto e otimizado a todos os dados estruturados de todos os 92 hotéis, com índices GIN pathops, generated columns, e full-text search em português.

- [ ] AC1: Script `data/scripts/ingest-enrichment.ts` lê todos os `data/enrichment/*.json` e mapeia para o JSONB `data` da tabela `hotels` seguindo a taxonomia de 58 campos
- [ ] AC2: Mapeamento JSON → JSONB respeita a estrutura da taxonomia (10 categorias, nomes de campos normalizados)
- [ ] AC3: Campos booleanos extraídos do JSON enriquecido: `pet_friendly` (de `pet_policy`), `pool_heated` (de `pool_description`/`pool_heated`), `bradesco_coupon` (de `_meta.bradesco_coupon`)
- [ ] AC4: Migração SQL cria índice GIN pathops: `CREATE INDEX idx_hotels_data_pathops ON hotels USING GIN (data jsonb_path_ops)`
- [ ] AC5: Migração SQL cria generated columns para campos de alta frequência:
  - `lodging_type text GENERATED ALWAYS AS (data->>'lodging_type') STORED`
  - `star_rating int GENERATED ALWAYS AS ((data->>'star_rating')::int) STORED`
  - `price_range text GENERATED ALWAYS AS (data->>'price_range') STORED`
- [ ] AC6: Migração SQL adiciona `tsvector` para full-text search em português:
  - `search_vector tsvector GENERATED ALWAYS AS (to_tsvector('portuguese', coalesce(name,'') || ' ' || coalesce(data->>'description','') || ' ' || coalesce(data->>'differentials',''))) STORED`
  - Índice GIN no `search_vector`
- [ ] AC7: Tool `query_hotel_details` implementada em `backend/src/tools/query-hotel-details.ts`:
  - Parâmetros: `hotelSlug` (required), `category` (optional: identity, accommodations, infrastructure, gastronomy, policies, transport, experiences, reputation, concierge, integration)
  - Se `category` informada: retorna apenas dados daquela categoria do JSONB
  - Se `category` omitida: retorna resumo geral (campos principais de cada categoria)
  - Fallback: se campo não existe, retorna `null` (nunca inventa)
- [ ] AC8: Tool `query_hotel_details` registrada no agent toolset da Stella
- [ ] AC9: Script de ingestão é idempotente (roda múltiplas vezes sem duplicar/perder dados, usa merge strategy no JSONB)
- [ ] AC10: RLS policy garante que Stella (service role) só tem READ access nas tabelas `hotels`
- [ ] AC11: Após ingestão, verificação automática: `SELECT COUNT(*) FROM hotels WHERE data->>'description' IS NOT NULL` = 92 (ou próximo)
- [ ] AC12: Script gera relatório de cobertura: quantos hotéis por campo, campos com < 50% cobertura listados para ação futura

## Decisões Técnicas (@architect + @data-engineer)

### Indexação JSONB
- **GIN pathops** (não default) — 3x menor, cobre 90% dos padrões com operador `@>`
- **Generated columns** para campos de alta frequência de query — auto-sync com JSONB
- **Full-text search** em português como fallback quando RAG tem baixa confiança

### Estratégia de Merge
- JSONB merge com `||` operator no PostgreSQL (deep merge)
- Preserva dados existentes do XLSX ingest
- Adiciona/sobrescreve com dados enriquecidos
- Metadados de ingestão: `_enrichment_version`, `_enriched_at`, `_sources`

### Caching
- A 92 hotéis, **não usar Redis** — carregar sumários na sessão da Stella
- Full JSONB fetch on-demand por hotel (Supabase connection pooler é suficiente)
- Revisitar se catálogo ultrapassar 500 hotéis

### RLS
- Service role: READ-only em `hotels` e `faq_embeddings`
- Anon role: sem acesso
- Admin role: CRUD completo

### Mapeamento JSON → Taxonomia (FIELD_MAP)

O JSON enriquecido usa flat keys (`deep_enrichment.*`) que devem ser remapeados para a estrutura da taxonomia (10 categorias). O script deve ter um `FIELD_MAP` explícito. Exemplo para a categoria **Gastronomia**:

```typescript
// Exemplo de FIELD_MAP parcial — categoria 4 (Gastronomia)
const FIELD_MAP: Record<string, string> = {
  // deep_enrichment key → taxonomia JSONB path
  'restaurants':           'gastronomy.restaurants',        // 4.1
  'meals_included':        'gastronomy.meals_included',     // 4.2
  'special_dining':        'gastronomy.special_dining',     // 4.3
  'room_service':          'gastronomy.room_service',       // 4.4
  'dietary_options':       'gastronomy.dietary_options',     // 4.5

  // categoria 1 (Identidade)
  'website_url':           'identity.website',              // 1.7
  'google_maps_url':       'identity.google_maps',          // 1.8
  'description':           'identity.description',          // 1.9
  'lodging_type':          'identity.lodging_type',         // 1.10

  // ... demais categorias seguem o mesmo padrão
  // Referência completa: docs/architecture/HOTEL-DATA-TAXONOMY.md
};
```

O dev deve completar o FIELD_MAP para todas as 10 categorias consultando a taxonomia.

### Tratamento de nulls

Campos `null` no JSON não devem sobrescrever dados existentes no JSONB. Usar strategy: "merge non-null only".

### Validação de dados

Antes de inserir, validar tipos (booleanos devem ser boolean, arrays devem ser arrays). Log de warnings para dados com tipo incorreto.

### Cenários de Teste

| # | Cenário | Input | Expected |
|---|---|---|---|
| CT-1 | Hotel com todos os campos | JSON com ~50 campos preenchidos | JSONB populado com 10 categorias, booleanos corretos |
| CT-2 | Hotel com muitos nulls | JSON com ~20 campos null | Apenas campos non-null inseridos, nulls ignorados |
| CT-3 | Hotel sem enrichment | Hotel que existe no banco mas não tem JSON em `data/enrichment/` | JSONB inalterado (mantém `{source: 'xlsx-ingest'}`) |
| CT-4 | Re-execução (idempotência) | Rodar script 2x seguidas | Dados idênticos, sem duplicação, timestamps atualizados |
| CT-5 | Booleano ambíguo | `pet_policy: "Aceita pets de pequeno porte com taxa de R$50"` | `pet_friendly: true` (extração de boolean de texto) |

**Dados de teste:** Usar os JSONs reais de `data/enrichment/` — são dados de produção e representam todos os cenários acima naturalmente (hotéis com cobertura alta como `fasano-rj.json` e hotéis com campos faltantes).
