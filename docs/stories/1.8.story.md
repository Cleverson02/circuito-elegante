## Descrição

Implementar pipeline de vetorização que converte TODO o conteúdo rico dos 92 hotéis (não apenas FAQs) em embeddings categorizados no pgvector, ativando a busca semântica (RAG) da Stella para qualquer pergunta aberta sobre qualquer hotel.

**Problema/Necessidade:** A tabela `faq_embeddings` está 100% vazia (0 rows). O pipeline RAG da Stella está completamente não-funcional — ela não consegue responder perguntas como "O hotel tem spa?", "Qual horário do check-in?", "O que fazer na região?". A infraestrutura (pgvector, chunker, embedding model, similarity search) já existe (Story 1.4), mas nunca foi populada.

**Visão:** Após esta story, a Stella responde qualquer pergunta aberta sobre qualquer hotel com base em dados reais vetorizados, com pre-filtering por hotel_id + category para máxima performance e relevância.

**Filosofia:** Vetorizar ALL rich content, não apenas FAQs. Um concierge de classe mundial precisa responder sobre gastronomia, atividades, políticas, localização — tudo. Categorizar embeddings permite pre-filtering cirúrgico.

## Decisões Técnicas (@architect + @data-engineer)

### Por que HNSW ao invés de IVFFlat?
- IVFFlat requer tuning de `lists` que é frágil em datasets pequenos
- HNSW com `m=16, ef_construction=64` é plug-and-play e mantém recall > 99%
- Performance excelente para o volume esperado (~2000-5000 embeddings)

### Por que embeddings categorizados?
- Pre-filtering por `hotel_id + category` antes do vector search reduz search space de 5000 para ~5-25 rows
- Isso é mais rápido e mais preciso que buscar em todo o dataset
- Permite à Stella direcionar: "pergunta sobre gastronomia" → busca só em `experience`

### Por que text-embedding-3-small?
- Recomendação do @data-engineer: não fazer upgrade para `large` — latência não justifica para 92 hotéis
- 1536 dimensões são suficientes para este domínio
- Custo-benefício ótimo

### Chunking: Unidade Lógica vs. Token Count
- Dados hoteleiros são naturalmente estruturados — cada campo é uma unidade de informação
- Chunking por token count quebraria "Restaurante Gero: culinária italiana, aberto 19h-23h" no meio
- 1 chunk = 1 unidade lógica = melhor retrieval

### Ordem de Processamento
1. Ler hotel do banco (JSONB `data` populado pela 1.7)
2. Passar pelo JSON-to-Markdown adapter (por categoria)
3. Cada documento Markdown → chunker (unidade lógica)
4. Cada chunk → embedding (text-embedding-3-small)
5. Upsert no `faq_embeddings` com category + metadata

### Rate Limiting
OpenAI embedding API tem rate limits. Usar batch de 100 (já configurado no `embedding.ts`). Processar hotéis sequencialmente, chunks em batch.

### Custo Estimado
~92 hotéis × 25 chunks × ~200 tokens = ~460K tokens ≈ $0.01 (text-embedding-3-small é $0.02/1M tokens)
