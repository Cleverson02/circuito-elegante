## Descrição

Implementar o método `registerCustomer()` no ElevareClient que chama `POST /customers` na API Elevare para registrar ou atualizar um hóspede antes de gerar a cotação. Este é o **segundo passo do Golden Discovery #2** — o fluxo mágico `search (3.1) → customer registration (3.2) → quotation (3.3)`.

**Problema/Necessidade:** A API Elevare exige um `customerId` para criar cotações (`POST /quotations`). Sem registrar o hóspede como customer na Elevare, não é possível gerar o link de pagamento. Os dados do hóspede já existem em `guest_profiles` (PostgreSQL) e na sessão Redis — esta story extrai, valida, mapeia para o formato Elevare e registra, retornando o `customerId` que será usado pela Story 3.3.

**Contexto de integração:** O Orchestrator Agent chama `register_customer` como tool durante o fluxo `API_BOOKING`. O tool extrai dados de `guest_profiles` (por phone) e da sessão ativa, mapeia para o payload Elevare, e cacheia o `customerId` no Redis para evitar chamadas duplicadas.

### Redis Caching Strategy

- **Key pattern:** `elevare_customer:{phone}` (E.164 format as key — unique per guest)
- **Value:** `customerId` string (e.g., `"CUST-789"`)
- **TTL:** 24h (86400s) — matches session TTL, ensures customer re-registration if guest returns after >24h
- **Rationale:** Avoids redundant `POST /customers` calls within the same session day. The Elevare API is idempotent, so duplicate calls are safe but wasteful.

### Field Mapping: guest_profiles → Elevare

| guest_profiles column | Elevare field | Notes |
|----------------------|---------------|-------|
| `name` | `name` | Required. If null in DB, reject with validation error |
| `phone_number` | `phone` | Required. E.164 format. Primary lookup key |
| `email` | `email` | Optional. Validate format if present |
| `language` | `language` | From session or guest_profiles. Default 'pt' |

### Idempotency Strategy

1. **Client-side (Redis cache):** Before calling API, check `elevare_customer:{phone}`. If cached, return immediately.
2. **Server-side (Elevare API):** The API itself is idempotent — same guest data returns the same `customerId`. This is the safety net if cache misses.
3. **Race condition:** Two concurrent requests for same guest — both may call API, both get same `customerId`, second SET to Redis is a no-op (same value). No harm.

### Pattern Reference: transfer-to-human.ts

The `transfer-to-human.ts` tool shows the pattern for extracting guest data in a tool context:
- Zod schema for parameters
- `tool()` from `@openai/agents` for SDK registration
- Build structured output from raw data
- Return typed response for Orchestrator consumption

### Error Handling Strategy

```typescript
// 4xx → Data validation issue (guest data malformed for Elevare)
if (status >= 400 && status < 500) {
  throw new ElevareCustomerValidationError(field, reason);
}
// 5xx / timeout → Infrastructure issue (retry handled by base client from 3.1)
if (status >= 500) {
  throw new ElevareApiError('POST /customers', status, body);
}
```

### Logging — PII Masking

Phone masking: `+5521999998888` → `+55***8888` (show country code + last 4).
Email masking: `joao@email.com` → `j***@email.com` (show first char + domain).
Never log full name in production logs.
