## Descrição

Este story implementa o **Fallback & Recovery Elegante** — o sistema que transforma falhas técnicas da API Elevare em experiências de concierge premium. Quando o Elevare retorna timeout, 500, erro de rede ou qualquer falha, o hóspede NUNCA vê "Erro", "Falha de conexão", "API indisponível" ou qualquer termo técnico. Em vez disso, Stella mascara a falha como um processo premium natural e, se necessário, realiza handover silencioso ao Chatwoot com contexto completo.

**Filosofia — Graceful Degradation (PRD §3.3):** Na hotelaria de alto padrão, falhas são invisíveis. Um concierge humano de luxo nunca diria "nosso sistema caiu" — ele diria "permita-me verificar com nossa equipe de reservas, um de nossos especialistas dará continuidade ao seu atendimento em instantes". Stella replica exatamente essa experiência. A falha técnica vira uma transição elegante para atendimento humano especializado, mantendo a percepção de cuidado contínuo.

**Problema/Necessidade:** O `ElevareClient` (Story 3.1) implementa circuit breaker, retry e timeout — mas quando todas essas defesas falham, o erro sobe sem tratamento. Sem o módulo de fallback, uma exceção `ElevareApiError` ou `ElevareTimeoutError` resultaria numa mensagem genérica de erro exposta ao hóspede via WhatsApp, quebrando a ilusão de concierge premium e potencialmente assustando o cliente. O fallback é a última linha de defesa entre a falha técnica e a experiência do hóspede.

**Decisão Arquitetural:** O fallback é implementado como um **serviço stateless** que recebe o erro, o estado da sessão e o idioma, e retorna uma tupla `{ guestMessage, handoverTriggered, sentryEventId }`. O serviço: (1) classifica o erro por tipo (timeout, API error, network error, circuit open), (2) tenta 1 retry automático para erros transientes (5xx) antes de acionar o fallback, (3) seleciona a mensagem elegante no idioma correto, (4) dispara handover silencioso ao Chatwoot via `buildHandoverSummary()` existente, (5) reporta ao Sentry com contexto completo. O módulo é uma camada acima do `ElevareClient` — quem chama o client pode envolver a chamada num `try/catch` e delegar ao fallback.

**Fluxo:**
```
Orchestrator chama ElevareClient.search() ou .quotation()
  → Retry interno do ElevareClient (até 3x para 5xx)
  → Todas as tentativas falham
  → catch(error)
  → handleElevareFailure(error, sessionData, language)
    → classifyError(error) → 'timeout' | 'api_error' | 'network_error' | 'circuit_open'
    → shouldRetryOnce(error) → true para 5xx não-retry-exausted
      → Retry 1x → sucesso? return resultado normal
      → Falha? continua fallback
    → selectFallbackMessage(errorType, language) → mensagem elegante em PT/EN/ES
    → triggerSilentHandover(sessionData, error) → buildHandoverSummary() + Chatwoot
    → reportToSentry(error, sessionData) → eventId
    → return { guestMessage, handoverTriggered: true, sentryEventId }
```

## Dev Notes

### Error Classification — Deep Dive

O `classifyError()` é a peça central do fallback. Ele transforma um `Error` genérico numa das 4 categorias que determinam tanto a mensagem ao hóspede quanto a decisão de handover.

**Hierarquia de classificação (ordem de prioridade):**

```typescript
function classifyError(error: unknown): ElevareErrorType {
  // 1. Circuit breaker aberto — mais específico, check primeiro
  if (error instanceof ElevareCircuitOpenError) return 'circuit_open';

  // 2. Timeout — segundo mais específico
  if (error instanceof ElevareTimeoutError) return 'timeout';
  if (error instanceof DOMException && error.name === 'AbortError') return 'timeout';

  // 3. API error (4xx/5xx) — check status code
  if (error instanceof ElevareApiError) return 'api_error';

  // 4. Network errors — check error code
  if (error instanceof Error) {
    const networkCodes = ['ECONNRESET', 'ECONNREFUSED', 'ETIMEDOUT'];
    if ('code' in error && networkCodes.includes((error as any).code)) {
      return 'network_error';
    }
    if (error.message.includes('fetch failed')) return 'network_error';
  }

  // 5. Default — classificar como api_error (aciona handover — seguro)
  return 'api_error';
}
```

**Por que `api_error` é o default:** Na dúvida, é melhor acionar o handover (seguro) do que deixar o hóspede sem atendimento. Um erro desconhecido classificado como `api_error` gera handover imediato — que é o comportamento mais seguro.

### Sentry Integration Pattern

```typescript
import * as Sentry from '@sentry/node';

function reportToSentry(
  error: unknown,
  sessionData: SessionData,
  errorType: ElevareErrorType,
  language: string,
): string {
  // Sentry pode não estar configurado em dev local
  if (!Sentry.getCurrentHub().getClient()) {
    logger.info('sentry_not_configured', { message: 'Skipping Sentry report in dev mode' });
    return '';
  }

  let eventId = '';

  Sentry.withScope((scope) => {
    // Fingerprinting — agrupa por tipo de erro para evitar flooding
    scope.setFingerprint(['elevare-fallback', errorType]);

    // Tags para filtragem no dashboard
    scope.setTag('module', 'elevare-fallback');
    scope.setTag('errorType', errorType);
    scope.setTag('language', language);

    // Contexto extra — visível no evento individual
    scope.setExtra('sessionId', sessionData.hotelId ? `${sessionData.hotelId}:${sessionData.guestPhone}` : 'unknown');
    scope.setExtra('errorType', errorType);
    scope.setExtra('language', language);
    scope.setExtra('hotelFocus', sessionData.hotelId);
    scope.setExtra('lastIntent', sessionData.context?.lastIntent ?? null);
    scope.setExtra('agentState', sessionData.agentState ?? null);

    // Capturar exceção
    eventId = Sentry.captureException(error instanceof Error ? error : new Error(String(error)));
  });

  return eventId;
}
```

**Fingerprinting:** O `scope.setFingerprint(['elevare-fallback', errorType])` garante que 1000 timeouts geram 1 issue no Sentry (com 1000 events), não 1000 issues separados. Isso é crítico para evitar alert fatigue.

### Chatwoot Handover — Context Preservation

O handover silencioso é a parte mais importante do fallback. Quando o hóspede é transferido para o Chatwoot, o atendente humano deve receber TUDO que o hóspede discutiu — sem perder contexto.

```typescript
function triggerSilentHandover(
  sessionData: SessionData,
  error: unknown,
  errorType: ElevareErrorType,
): HandoverSummary {
  const summary = buildHandoverSummary({
    reason: 'api_failure',
    guestName: (sessionData.context?.guestName as string) ?? null,
    guestPhone: sessionData.guestPhone,
    hotelFocus: sessionData.hotelId,
    conversationSummary: buildConversationSummary(sessionData),
    lastIntent: (sessionData.context?.lastIntent as string) ?? null,
    preferences: (sessionData.context?.preferences as Record<string, unknown>) ?? {},
  });

  // Enriquecer com contexto de erro (para o atendente, não para o hóspede)
  const enrichedSummary = {
    ...summary,
    errorContext: {
      errorType,
      originalError: error instanceof Error ? error.message : String(error),
      timestamp: new Date().toISOString(),
      sessionId: `${sessionData.hotelId}:${sessionData.guestPhone}`,
      lastElevareEndpoint: detectLastEndpoint(error),
    },
  };

  logger.info('silent_handover_triggered', {
    sessionId: enrichedSummary.errorContext.sessionId,
    errorType,
    reason: 'api_failure',
  });

  return enrichedSummary;
}
```

**`buildConversationSummary(sessionData)`** — compila o contexto da sessão num resumo legível para o atendente humano:
```typescript
function buildConversationSummary(sessionData: SessionData): string {
  const parts: string[] = [];
  parts.push(`Hotel: ${sessionData.hotelId}`);
  parts.push(`Idioma: ${sessionData.language}`);
  if (sessionData.context?.lastIntent) {
    parts.push(`Última intenção: ${sessionData.context.lastIntent}`);
  }
  if (sessionData.agentState) {
    parts.push(`Estado do agente: ${sessionData.agentState}`);
  }
  // Adicionar qualquer dado relevante do contexto
  const ctx = sessionData.context ?? {};
  if (ctx.checkIn) parts.push(`Check-in: ${ctx.checkIn}`);
  if (ctx.checkOut) parts.push(`Check-out: ${ctx.checkOut}`);
  if (ctx.adults) parts.push(`Adultos: ${ctx.adults}`);
  if (ctx.children) parts.push(`Crianças: ${ctx.children}`);
  return parts.join(' | ');
}
```

### Retry Logic — One-Shot Fallback Retry

O fallback faz **no máximo 1 retry** para erros transientes. Isso é diferente do retry do `ElevareClient` (que faz até 3):

```
ElevareClient retry (3x) → falha → erro sobe → handleElevareFailure
  → classifyError → api_error (5xx)
  → isTransient5xx? → SIM
  → retryFn fornecido? → SIM
  → tenta 1x mais → sucesso? → retorna resultado normal
                   → falha?  → guestMessage + handover + Sentry
```

**Por que 1 retry adicional?** O `ElevareClient` faz retry interno, mas há cenários onde o erro sobe antes de esgotar os retries (ex: circuit breaker abre no meio). O fallback dá uma última chance antes de acionar o handover.

**Implementação:**
```typescript
async function handleElevareFailure(
  error: unknown,
  sessionData: SessionData,
  language: string,
  retryFn?: () => Promise<unknown>,
): Promise<FallbackResult> {
  const errorType = classifyError(error);
  const validLanguage = validateLanguage(language);

  // Retry para erros transientes 5xx (1 tentativa)
  if (errorType === 'api_error' && isTransient5xx(error) && retryFn) {
    try {
      const result = await retryFn();
      logger.info('fallback_retry_succeeded', { sessionId: sessionData.guestPhone });
      return { guestMessage: '', handoverTriggered: false, sentryEventId: '', retryResult: result };
    } catch (retryError) {
      logger.warn('fallback_retry_failed', {
        sessionId: sessionData.guestPhone,
        originalError: errorType,
      });
      // Continua com fallback completo usando o novo erro
      error = retryError;
    }
  }

  // Selecionar mensagem elegante
  const guestMessage = selectFallbackMessage(errorType, validLanguage);

  // Determinar se handover é necessário
  const handoverTriggered = errorType !== 'timeout';

  // Handover silencioso ao Chatwoot
  let handoverSummary: HandoverSummary | null = null;
  if (handoverTriggered) {
    handoverSummary = triggerSilentHandover(sessionData, error, errorType);
  }

  // Reportar ao Sentry
  const sentryEventId = reportToSentry(error, sessionData, errorType, validLanguage);

  // Logging para analytics
  logger.warn('elevare_fallback_triggered', {
    errorType,
    language: validLanguage,
    sessionId: sessionData.guestPhone,
    hotelId: sessionData.hotelId,
    handoverTriggered,
    sentryEventId,
  });

  return { guestMessage, handoverTriggered, sentryEventId };
}
```

### `isTransient5xx()` Helper

```typescript
function isTransient5xx(error: unknown): boolean {
  if (error instanceof ElevareApiError) {
    return error.statusCode >= 500 && error.statusCode < 600;
  }
  return false;
}
```

Retorna `true` apenas para status 500-599. Erros 4xx (400, 401, 403, 404, 422) NUNCA são retried — são erros de client que não mudam com retry.

### Circuit Breaker Interaction (Story 3.1)

O fallback trabalha ACIMA do circuit breaker do `ElevareClient`:

```
Request → ElevareClient
           ├── Circuit CLOSED → faz request → retry se 5xx (até 3x)
           ├── Circuit HALF_OPEN → faz 1 request de teste
           └── Circuit OPEN → throw ElevareCircuitOpenError imediatamente
                                   ↓
                          handleElevareFailure()
                          classifyError() → 'circuit_open'
                          → mensagem elegante de reservas
                          → handover imediato
                          → Sentry report
```

Quando o circuit breaker está OPEN, não há sentido em retry — o `ElevareClient` nem faz a request HTTP. O fallback reconhece isso (`circuit_open`) e aciona handover imediato com a mensagem mais elaborada: "Deixa eu verificar com nossa equipe de reservas..."

### Example Fallback Flow — Complete Walk-Through

**Cenário:** Hóspede brasileiro busca quartos no Hotel ABC. Elevare retorna 503 três vezes (retry do client esgotado).

```
1. Orchestrator: ElevareClient.search({ hotelId: 'abc', checkIn: '2026-05-01', ... })
2. ElevareClient: GET /search → 503 (retry 1)
3. ElevareClient: GET /search → 503 (retry 2)
4. ElevareClient: GET /search → 503 (retry 3 — último)
5. ElevareClient: throw ElevareApiError({ statusCode: 503, retryCount: 3 })
6. Orchestrator: catch(error) → handleElevareFailure(error, session, 'pt')
7. Fallback: classifyError(error) → 'api_error'
8. Fallback: isTransient5xx(error) → true, retryFn fornecido
9. Fallback: retry 1x → 503 novamente → falha
10. Fallback: selectFallbackMessage('api_error', 'pt')
    → "Vou conectar você com um de nossos especialistas que pode ajudar de forma mais ágil."
11. Fallback: handoverTriggered = true (api_error → handover imediato)
12. Fallback: triggerSilentHandover(session, error, 'api_error')
    → buildHandoverSummary({ reason: 'api_failure', guestName: 'João', ... })
    → enrichedSummary com errorContext
13. Fallback: reportToSentry(error, session, 'api_error', 'pt')
    → fingerprint: ['elevare-fallback', 'api_error']
    → eventId: 'evt_abc123'
14. Fallback: logger.warn('elevare_fallback_triggered', { ... })
15. Fallback: return {
      guestMessage: "Vou conectar você com um de nossos especialistas...",
      handoverTriggered: true,
      sentryEventId: 'evt_abc123'
    }
16. Orchestrator: envia guestMessage ao hóspede via WhatsApp
17. Hóspede vê: "Vou conectar você com um de nossos especialistas que pode ajudar de forma mais ágil."
18. Chatwoot: atendente recebe HandoverSummary com todo contexto + errorContext
```

**Resultado:** Hóspede acha que está sendo transferido para um especialista (que ele é). Não sabe que houve falha técnica.

### Anti-Patterns to AVOID

| Anti-Pattern | Por Que Está Errado | O Que Fazer |
|-------------|--------------------|----|
| `"Erro: API Elevare retornou 503"` | Expõe detalhes técnicos ao hóspede | Usar mensagem elegante do dicionário i18n |
| `"Desculpe pelo inconveniente"` | Admite falha — hóspede não precisa saber que houve falha | Mensagem como ação natural: "Vou conectar você com um especialista..." |
| Retry infinito no fallback | Atrasa resposta indefinidamente | Máximo 1 retry, depois fallback imediato |
| Logar API key no Sentry extra | Vazamento de credencial | NUNCA incluir ELEVARE_API_KEY em logs ou Sentry |
| Swallow error sem Sentry | Bug invisível — equipe nunca descobre o problema | SEMPRE reportar ao Sentry, mesmo que o hóspede não veja |
| Handover sem contexto | Atendente humano recomeça do zero — péssima experiência | `buildHandoverSummary()` com TUDO que o hóspede discutiu |
| Mensagem genérica para todos os erros | Perde nuance — timeout pode resolver sozinho, circuit_open não | Mensagem específica por tipo de erro |
| `guestMessage` em inglês para hóspede brasileiro | Quebra experiência | i18n com fallback para PT (mercado principal) |
