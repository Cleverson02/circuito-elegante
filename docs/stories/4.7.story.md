## Descricao

Este story implementa o **fallback de fora de expediente** — a experiencia que Stella oferece quando um hospede entra em contato fora do horario operacional (antes das 8h ou apos as 22h, horario de Brasilia).

**Problema/Necessidade:** Hospedes de luxo viajam internacionalmente e podem estar em fusos diferentes. Um hospede em Paris (5h a frente de BRT) pode enviar mensagem as 2h da manha BRT. Sem fallback, a mensagem entraria no pipeline normal e o hospede receberia curadoria completa — mas nao haveria humano disponivel para atender se surgir uma necessidade que a Stella nao consiga resolver. O fallback garante que: (1) o hospede recebe uma mensagem elegante imediata, (2) um lead e registrado para follow-up no primeiro horario util.

**Comportamento:**
```
Hospede envia mensagem fora do expediente (ex: 23:45 BRT)
  ↓
BusinessHoursGuard intercepta ANTES do pipeline de agentes
  ↓
Stella responde: mensagem elegante + promessa de follow-up
  ↓
Lead registrado: { phone, message, timestamp, scheduledFollowUp }
  ↓
No proximo horario util (8h), template WhatsApp 'out_of_hours_followup' (Story 4.9) pode ser usado
```

- [x] AC4: Mensagem de fallback WhatsApp: texto elegante em portugues informando que o atendimento esta fora do expediente, prometendo follow-up no primeiro horario util. Tom: caloroso, profissional, luxury. Exemplo: `"Boa noite! Obrigada por entrar em contato com o Circuito Elegante. Nosso atendimento personalizado funciona das 8h as 22h. Registrei sua mensagem e retornaremos no primeiro horario util. Ate breve! ✨"`. Mensagem configuravel via constante exportada.
- [x] AC5: Mensagem de fallback Website: mesma mensagem adaptada para widget (sem emojis WhatsApp). Enviada via WebSocket.
- [x] AC6: `registerOutOfHoursLead(phone: string, message: string, channel: 'whatsapp' | 'website', sessionId: string): Promise<void>` em `backend/src/services/business-hours.ts` que persiste o lead. Storage: append a um arquivo JSON local `data/out-of-hours-leads.json` (MVP — futuro: Supabase).
- [x] AC7: Cada lead registrado contem: `{ phone, message, channel, sessionId, receivedAt: ISO timestamp, scheduledFollowUp: proximo dia util 08:00 BRT, status: 'pending' }`.
- [x] AC8: Log estruturado: `{ event: 'out_of_hours_intercepted', phone, channel, sessionId, currentHourBRT, message }` quando intercepta, e `{ event: 'out_of_hours_lead_registered', phone, leadId }` quando registra lead.
- [x] AC9: Se a verificacao de horario falhar (erro de timezone, excecao), **nao bloquear** — seguir fluxo normal do pipeline (fail-open). Log error `business_hours_check_failed`.
- [x] AC10: Testes unitarios: (a) isWithinBusinessHours retorna true as 10h BRT, false as 23h BRT, (b) boundary: true as 08:00, true as 21:59, false as 22:00, false as 07:59, (c) lead registrado com campos corretos, (d) fail-open quando timezone invalido — cobertura >= 90% branches.

## Dev Notes

### Previous Story Insights
- **Story 4.2 (Buffer):** buffer-processor.ts e o ponto de entrada apos consolidacao WhatsApp. Guard de horario deve ser o primeiro check antes do pipeline. [Source: buffer-processor.ts:65-106]
- **Story 4.9 (Templates):** Template `out_of_hours_followup` sera usado para follow-up automatico. Story 4.7 registra o lead; Story 4.9 define o template para envio futuro.

### Technical Constraints
- **Intl.DateTimeFormat** para timezone BRT — nao usar bibliotecas externas (moment, luxon) para MVP. Node.js 18+ suporta timezone nativamente.
- **Storage local JSON** — MVP com arquivo JSON. Futuro: tabela Supabase `out_of_hours_leads`.
- **Fail-open:** Se qualquer erro ocorrer na verificacao de horario, o hospede NAO deve ser bloqueado — seguir pipeline normal. Melhor atender fora de hora do que nao atender.
- **Tom luxury:** A mensagem de fallback deve transmitir exclusividade e cuidado, nao um generico "estamos fechados".

### Integracao com Stories Subsequentes
- **Story 4.9 (Templates):** Consome o lead registrado para enviar template `out_of_hours_followup` no proximo horario util
- **Story 4.8 (E2E Tests):** Testa o fluxo fora-de-expediente end-to-end
