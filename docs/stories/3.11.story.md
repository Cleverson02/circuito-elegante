## Descricao

Auditoria conduzida em 2026-04-16 comparando `backend/src/integrations/elevare/*` com a Postman collection oficial (`docs/api/elevare-global-agent.postman_collection.json`) identificou **divergencias criticas** que tornam a integracao nao funcional contra a API real.

**Resumo do diagnostico:**

1. **Auth default errada.** Codigo envia header `X-Api-Key` — esse header nao existe na API da Elevare. A API real exige DOIS headers: `x-client-id` + `x-client-secret` em toda requisicao. A variavel `ELEVARE_API_KEY` no `.env.example` e `config/env.ts` e uma var fantasma (valor ignorado pela API).

2. **Prefixo de rota ausente.** Todos os endpoints (search, multi-search, customers, quotations) precisam prefixo `/global-agent/`. Codigo atual chama `/search`, `/customers`, etc — retorna 404.

3. **Query params renomeados.** Endpoint `search` usa nomes diferentes: `q`, `CheckIn`, `CheckOut`, `ad`, `ch`, `ag` (nao `hotelId`/`checkIn`/`adults`/`children` como o codigo envia).

4. **Payload de customers incorreto.** Postman exige `{primaryPhone, firstName, lastName, email, cpf?, birthDate?, ...}`. Codigo envia `{name, phone, email, language}` — `language` nao existe no contrato, `name` precisa split em first/last, `phone` precisa rename pra `primaryPhone`.

5. **Env schema permissivo demais.** `ELEVARE_CLIENT_ID` e `ELEVARE_CLIENT_SECRET` estao marcados como optional em `config/env.ts` quando deveriam ser obrigatorios (a API real falha sem eles). `ELEVARE_WEBHOOK_SECRET` nao tem entrada no schema zod.

6. **Base URL de producao incorreta.** Default hardcoded do codigo usa `https://api.elevare.com.br/v1`. A URL oficial correta (confirmada pelo usuario) e `https://api.elevare.com.br/api/v1` — falta o segmento `/api`.

**Tres pontos ficam documentados mas NAO sao corrigidos nesta story** (dependem de resposta da Elevare — viram Story 3.12 quando confirmados):
- Schema real dos webhooks inbound (Elevare → Stella). Os event types que Stella processa (`quote_expiring`, `reservation_confirmed`, `payment_failed`, `quote_created`) nao aparecem na Postman.
- Busca de reservas por `guestPhone` — Postman so documenta `email`/`confirmationNumber`/`guestName`.
- Tipo de `offerId`: `types.ts:39` declara `string` em `ElevareOffer`, mas quotation body usa `number`.

**Impacto se nao corrigido:** Fluxo de reserva nao funciona. Qualquer chamada a `ElevareClient.search()`, `.createCustomer()` ou `.createQuotation()` contra a API real resulta em 401 (auth) ou 404 (path). Stella nao consegue cotar reservas.

## Acceptance Criteria

- [x] **AC1 — Base URL:** `ELEVARE_API_URL` nao tem default hardcoded em `config.ts` — forca definicao explicita via env. `.env.example` comenta a URL oficial: `https://api.elevare.com.br/api/v1`.
- [x] **AC2 — Auth default em `client.ts`:** toda requisicao envia os headers `x-client-id` e `x-client-secret` (lidos de `ELEVARE_CLIENT_ID` e `ELEVARE_CLIENT_SECRET`). Header `X-Api-Key` removido. Flag `overrideAuth` do `reservations.ts` pode ser removida (nao e mais excecao).
- [x] **AC3 — Remocao de `ELEVARE_API_KEY`:** variavel removida de `.env.example`, `config/env.ts` (schema zod), `backend/src/integrations/elevare/config.ts`, `backend/src/integrations/elevare/client.ts` e de qualquer outro lugar onde apareca. `grep -r "ELEVARE_API_KEY" backend/ config/ .env.example tests/` retorna zero resultados.
- [x] **AC4 — Env schema obrigatorio:** `config/env.ts` marca `ELEVARE_CLIENT_ID` e `ELEVARE_CLIENT_SECRET` como `z.string().min(1)` (nao optional). `ELEVARE_WEBHOOK_SECRET` adicionada ao schema como `z.string().min(1)`. Falta de qualquer uma dessas tres vars faz o app falhar no boot com mensagem clara.
- [x] **AC5 — Prefixo `/global-agent/` em todas as rotas:** `search` → `/global-agent/search`, `multi-search` → `/global-agent/multi-search`, `customers` → `/global-agent/customers`, `quotations` → `/global-agent/quotations`, `quotations/{id}/payment-link` → `/global-agent/quotations/{id}/payment-link`, `quotations/{id}/extend` → `/global-agent/quotations/{id}/extend`.
- [x] **AC6 — Query params corretos em `search()`:** request envia `q={hotelId}`, `CheckIn={ISO}`, `CheckOut={ISO}`, `ad={adults}`, `ch={children}`, `ag={ages}` (nao `hotelId`/`checkIn`/`adults`/`children`/`childrenAges`). Se aplicavel, tambem ajustar em `multiSearch()` conforme Postman.
- [x] **AC7 — Payload de `customers` POST:** body envia `{primaryPhone, firstName, lastName, email, cpf?, birthDate?, ...}` conforme Postman. Codigo faz split de `name` em `firstName`/`lastName` (se input vier consolidado), rename `phone` → `primaryPhone`, remove campo `language`.
- [x] **AC8 — Testes unitarios atualizados:** `tests/unit/elevare-client.test.ts`, `tests/unit/elevare-customers.test.ts`, `tests/unit/elevare-quotations.test.ts` e `tests/unit/elevare-webhook.test.ts` refletem os novos paths (`/global-agent/*`), auth (`x-client-id`/`x-client-secret`) e payloads. Todos passam com `npm test:unit`.
- [x] **AC9 — Readiness verde:** `npm run readiness` retorna `✅ Ambiente pronto para a Stella` com todas as vars Elevare obrigatorias presentes e API alcancavel (ou skip OPT se ainda nao configurada em dev).
- [x] **AC10 — Ambiguidades documentadas:** arquivo `docs/stories/3.11-followup-ambiguities.md` criado listando os 3 pontos pendentes de resposta da Elevare (webhook schema, guestPhone, offerId type) com placeholders para resposta. Nao bloqueia merge desta story.
