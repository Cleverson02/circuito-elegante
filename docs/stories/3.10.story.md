## Descricao

Este story implementa a **suite de testes de integracao abrangente** que valida o EPIC-3 inteiro — o Fluxo de Ouro — ponta a ponta. E o quality gate definitivo antes de considerar a integracao Elevare pronta para producao.

**Por que testes de integracao sao criticos para um sistema AAA:**

Testes unitarios validam funcoes isoladas. Testes E2E validam o sistema rodando contra infra real. Mas entre esses dois extremos existe uma zona perigosa: a integracao entre componentes. O ElevareClient fala com Redis, que alimenta a Curadoria, que produz OfferMaps consumidos pela Correfencia, que resolve offerIds usados pela Quotation, que gera links monitorados por Webhooks. Se qualquer "costura" entre esses modulos estiver errada, o hospede experimenta uma falha silenciosa — a pior categoria de bug em um sistema de reservas de luxo.

Esta suite testa cada costura. Usa a Postman collection real da Elevare como fixture de dados (garantindo fidelidade ao contrato da API), simula o servidor Elevare com respostas realistas via mock server, e valida cada caminho do sistema: happy path, correfencia, curadoria, upsell, webhooks, fallback, circuit breaker e edge cases.

**Filosofia de teste:** Cada teste e uma narrativa. O hospede chega, busca, seleciona, registra, recebe link de pagamento. O webhook confirma. Stella parabeniza. Essa e a historia que cada teste conta — e se qualquer capitulo falhar, o teste falha com diagnostico preciso de onde a costura se rompeu.

**Decisao Arquitetural:** Os testes de integracao vivem em `tests/integration/` (separados dos `tests/unit/` e `tests/e2e/`). Usam um mock server (nock ou MSW) para simular a API Elevare, Redis real (test instance) para validar estado, e Fastify `inject()` para HTTP sem levantar servidor. Cada teste limpa Redis e DB no `beforeEach` para isolamento total. A Postman collection em `tests/fixtures/elevare-postman.json` e a single source of truth para payloads, URLs e headers esperados.

**Cobertura FR completa:**
```
FR2  (search)           → Happy Path: search retorna opcoes
FR3  (quotation)        → Happy Path: quotation gera paymentLink
FR4  (status consulta)  → Status: hospede consulta reserva
FR16 (curadoria)        → Curadoria: price buckets, max 3-4 opcoes
FR18 (correfencia)      → Correfencia: "quero a segunda" → offerId
FR19 (upsell)           → Upsell: deteccao + one-shot enforcement
FR20 (fallback)         → Fallback: API down → resposta elegante
FR22 (fotos)            → Search: fotos no payload
FR24 (requestId opaco)  → Seguranca: requestId nunca em mensagens
FR25 (webhooks)         → Webhooks: todos os event types
FR29 (regen link)       → Quotation: regeneracao de link
FR30 (extend link)      → Quotation: extensao de validade
FR31 (multi-search)     → Search: multi-search por cidade
```

## Acceptance Criteria

- [ ] AC1: Postman collection salva em `tests/fixtures/elevare-postman.json` contendo todos os endpoints Elevare (`GET /search`, `GET /multi-search`, `POST /customers`, `POST /quotations`, `PUT /quotations/{id}/payment-link`, `PUT /quotations/{id}/extend`, `GET /reservations/{id}`) com exemplos de request/response realistas
- [ ] AC2: Mock server configurado em `tests/helpers/elevare-mock-server.ts` usando nock que intercepta todas as chamadas a `ELEVARE_API_URL` e retorna respostas baseadas nos fixtures da Postman collection — setup em `beforeAll`, teardown em `afterAll`
- [ ] AC3: Response factories em `tests/fixtures/elevare-responses.ts` geram payloads Elevare realistas e parametrizaveis: `makeSearchResponse(overrides?)`, `makeCustomerResponse(overrides?)`, `makeQuotationResponse(overrides?)`, `makeReservationResponse(overrides?)`, `makeWebhookPayload(eventType, overrides?)`
- [ ] AC4: **Happy Path E2E** — teste valida o fluxo completo: `search(params)` retorna opcoes com fotos → curadoria seleciona 3-4 opcoes → hospede seleciona uma → `registerCustomer(data)` retorna customerId → `createQuotation(requestId, offerId, customerId)` retorna paymentLink — todas as chamadas intermediarias verificadas via Redis state
- [ ] AC5: **Correfencia** — teste envia "quero a segunda" apos curadoria apresentar 3 opcoes, verifica que `CoreferenceResolver` resolve para o `offerId` na posicao 2 do `OfferMap` armazenado em Redis
- [ ] AC6: **Curadoria** — teste envia 15 opcoes do search mock e verifica: (a) curadoria retorna exatamente 3-4 opcoes, (b) price buckets representados (budget, mid, premium), (c) opcoes com Bradesco coupon priorizadas quando `bradescoCoupon: true` no search original
- [ ] AC7: **Upsell** — teste verifica: (a) `detectUpsell` identifica upgrade dentro do threshold quando existe, (b) apos upsell oferecido, `shouldOfferUpsell` retorna `false` (one-shot enforcement via Redis), (c) upsell NAO detectado quando todas as opcoes tem o mesmo preco
- [ ] AC8: **Webhook — reservation_confirmed** — teste envia POST para `/webhooks/elevare` com evento `reservation_confirmed`, verifica: (a) evento armazenado em `webhook_events` (PostgreSQL), (b) response 200 retornado, (c) follow-up message gerada corretamente
- [ ] AC9: **Webhook — payment_failed** — teste envia evento `payment_failed`, verifica: (a) evento armazenado, (b) retry link gerado via `PUT /quotations/{id}/payment-link`, (c) mensagem elegante de retry enviada ao hospede
- [ ] AC10: **Webhook — quote_expiring** — teste envia evento `quote_expiring`, verifica: (a) evento armazenado, (b) notificacao sutil enviada ao hospede, (c) extensao de validade via `PUT /quotations/{id}/extend` quando aplicavel
- [ ] AC11: **Webhook Deduplicacao** — teste envia o MESMO evento duas vezes com o mesmo `eventId`, verifica: (a) primeiro processado normalmente (201), (b) segundo retorna 200 sem reprocessar (idempotente), (c) apenas 1 registro em `webhook_events`
- [ ] AC12: **Webhook Signature Validation** — teste envia webhook com assinatura invalida, verifica rejeicao com 401 Unauthorized e evento NAO armazenado
- [ ] AC13: **Fallback — API Timeout** — mock server configurado com delay > 8s, verifica: (a) circuit breaker abre apos threshold de falhas, (b) resposta elegante retornada ao hospede (nao "Error: ETIMEDOUT"), (c) handover para Chatwoot iniciado
- [ ] AC14: **Fallback — API 500** — mock server retorna 500 Internal Server Error, verifica: (a) retry logic executa (ate 3 tentativas), (b) apos retries esgotados, fallback elegante ativado, (c) log estruturado com contexto completo para Sentry
- [ ] AC15: **Fallback — Network Error** — mock server simula ECONNREFUSED, verifica: (a) circuit breaker registra falha, (b) fallback ativado imediatamente (sem retry para network error), (c) mensagem ao hospede em idioma da sessao (PT/EN/ES)
- [ ] AC16: **Circuit Breaker States** — teste valida transicoes: (a) CLOSED (operacao normal) → falhas consecutivas atingem threshold → OPEN (rejeita chamadas imediatamente), (b) OPEN → apos `resetTimeout` → HALF_OPEN (permite 1 chamada de teste), (c) HALF_OPEN → chamada de teste sucesso → CLOSED, (d) HALF_OPEN → chamada de teste falha → OPEN
- [ ] AC17: **Edge Case — Empty Search** — mock retorna `{ results: [] }`, verifica que curadoria retorna array vazio e mensagem elegante "Infelizmente nao encontramos opcoes para essas datas" (nao crash, nao mensagem generica)
- [ ] AC18: **Edge Case — Single Result** — mock retorna exatamente 1 opcao, verifica: (a) curadoria retorna 1 opcao (sem padding com duplicatas), (b) upsell retorna `{ detected: false }`, (c) correfencia funciona com mapa de 1 item
- [ ] AC19: **Edge Case — Expired requestId** — teste tenta criar quotation com requestId que nao existe mais no Redis (expirou), verifica: (a) erro capturado gracefully, (b) novo search sugerido ao hospede, (c) log de warning para analytics
- [ ] AC20: **Redis State Verification** — em CADA teste do happy path, apos cada etapa, verificar estado do Redis: (a) `request:{requestId}` existe apos search, (b) `offers:{sessionId}` existe apos curadoria, (c) `customer:{phone}` existe apos registration, (d) `quotation:{id}` existe apos quotation, (e) `upsell_offered:{sessionId}` existe apos upsell
- [ ] AC21: **Database State Verification** — testes de webhook verificam que `webhook_events` no PostgreSQL contem o registro correto com: `event_id`, `event_type`, `payload` (JSONB), `processed_at`, `created_at`
- [ ] AC22: **Test Isolation** — cada teste executa em isolamento total: (a) `beforeEach` limpa todas as chaves Redis de teste (`FLUSHDB` na DB de teste), (b) `beforeEach` faz rollback/truncate de tabelas de teste no PostgreSQL, (c) nenhum teste depende de estado de outro teste, (d) testes podem rodar em qualquer ordem
- [ ] AC23: **Coverage Threshold** — suite de integracao atinge >= 90% de cobertura de branches para os modulos em `backend/src/` que sao exercitados pelos testes (client, curadoria, correfencia, upsell, webhooks, fallback)
- [ ] AC24: **Determinism** — todos os testes passam de forma deterministica: (a) sem `setTimeout` arbitrarios (usar `jest.useFakeTimers()` para circuit breaker), (b) sem dependencia de timing real, (c) 100 execucoes consecutivas = 100 passes

## Scope

### IN
- `tests/integration/elevare-flow.test.ts` — Suite principal: happy path, curadoria, correfencia, upsell, edge cases
- `tests/integration/elevare-webhooks.test.ts` — Suite de webhooks: todos os event types, dedup, signature validation
- `tests/fixtures/elevare-postman.json` — Postman collection como fixture de dados (contratos da API)
- `tests/fixtures/elevare-responses.ts` — Factory functions para gerar payloads Elevare realistas
- `tests/helpers/elevare-mock-server.ts` — Mock server usando nock para interceptar chamadas Elevare
- Verificacao de estado Redis em cada etapa do fluxo
- Verificacao de estado PostgreSQL para webhook_events
- Circuit breaker state machine testing
- Fallback behavior testing (timeout, 500, network error)
- Coverage threshold enforcement (>= 90% para modulos integrados)

### OUT
- Testes contra a API Elevare real (staging ou producao) — exclusivamente mocks
- Testes de carga / performance / stress — escopo de NFR testing futuro
- Testes de UI / WhatsApp rendering — sem frontend neste story
- Testes de Persona Agent message generation (conteudo da mensagem) — apenas verificamos que a funcao foi chamada, nao o que ela produziu
- Mutacao de codigo dos modulos testados — este story testa, nao implementa
- Testes de deploy/rollback/infra — escopo @devops
- Monitoring/alerting setup — escopo operacional
- Testes de multi-tenancy (isolamento entre hoteis) — escopo futuro

## Dev Notes

### Mock Server Architecture — nock

**Por que nock e nao MSW:** nock opera no nivel de `http.request` do Node.js — intercepta ANTES de qualquer chamada de rede sair do processo. MSW opera no nivel de Service Worker (browser) ou `setupServer` (Node) com overhead maior. Para testes de integracao server-side (Fastify + ElevareClient), nock e mais leve, mais rapido e mais deterministic.

```typescript
// tests/helpers/elevare-mock-server.ts — Architecture

import nock from 'nock';

export function setupElevareMockServer(): void {
  // Disable real HTTP connections during tests
  nock.disableNetConnect();
  // Allow localhost for Fastify inject (not real HTTP, but safety net)
  nock.enableNetConnect('127.0.0.1');
}

export function teardownElevareMockServer(): void {
  nock.cleanAll();
  nock.enableNetConnect();
}

export function resetElevareMocks(): void {
  nock.cleanAll();
}

// --- Endpoint-specific mock helpers ---

export function mockSearchSuccess(response?: SearchResponse): nock.Scope {

    .get('/search')
    .query(true) // match any query params

export function mockSearchEmpty(): nock.Scope {
```

### Fastify Test Injection Pattern

**NAO levantar servidor HTTP para testes de integracao.** Usar `fastify.inject()`:

```typescript
import { buildApp } from '../../backend/src/server';

let app: FastifyInstance;

beforeAll(async () => {
  app = await buildApp({ logger: false }); // Disable logging in tests
  await app.ready();
});

afterAll(async () => {
  await app.close();
});

// Usage in tests:
const response = await app.inject({
  method: 'POST',
  url: '/webhooks/elevare',
  headers: {
    'content-type': 'application/json',
    'x-elevare-signature': 'valid-test-signature',
  },
  payload: makeWebhookPayload('reservation_confirmed'),
});

expect(response.statusCode).toBe(201);
expect(JSON.parse(response.body)).toMatchObject({ status: 'processed' });
```

**Vantagens do inject sobre Supertest:**
- Sem bind de porta → sem conflito entre testes paralelos
- Sem overhead de TCP → mais rapido
- Acesso direto ao Fastify lifecycle → melhor teardown
- Nao precisa `await app.listen()` — `app.ready()` basta

### Redis Test Setup/Teardown

```typescript
// tests/integration/setup.ts

import { createClient, type RedisClientType } from 'redis';

let redisClient: RedisClientType;

const REDIS_TEST_URL = process.env.REDIS_URL_TEST ?? 'redis://localhost:6379/1'; // DB 1 for tests!

beforeAll(async () => {
  redisClient = createClient({ url: REDIS_TEST_URL });
  await redisClient.connect();
});

beforeEach(async () => {
  // Nuclear cleanup — safe because we're on DB 1 (test)
  await redisClient.flushDb();
});

afterAll(async () => {
  await redisClient.flushDb(); // Final cleanup
  await redisClient.quit();
});

export { redisClient };
```

**CRITICO:** Usar Redis DB 1 (ou superior) para testes. NUNCA DB 0 (producao/dev). O `FLUSHDB` no `beforeEach` apaga TUDO na DB selecionada — se estiver em DB 0, destroi sessoes e dados de dev.

### Nock Best Practices para Este Projeto

| Pratica | Porque |
|---------|--------|
| `nock.disableNetConnect()` no `beforeAll` | Garante zero chamadas reais — qualquer chamada nao mockada falha loud |
| `nock.cleanAll()` no `beforeEach` | Remove interceptors pendentes de testes anteriores |
| `nock.isDone()` no final de testes criticos | Verifica que todos os interceptors configurados foram consumidos |
| `nock.enableNetConnect('127.0.0.1')` | Permite Fastify inject funcionar (mesmo que nao use HTTP real) |
| `.times(N)` para retry tests | Permite controlar exatamente quantas vezes o mock responde |
| `.delay(ms)` para timeout tests | Simula latencia sem `setTimeout` real |
| `nock.recorder.rec()` (dev only) | Grava chamadas reais para gerar fixtures — NAO commitar recordings |

### Anti-Patterns em Testes de Integracao

| Anti-Pattern | Por que e Errado | Faca Isso |
|-------------|------------------|-----------|
| `setTimeout(resolve, 1000)` | Indeterministic, flaky, lento | `jest.useFakeTimers()` + `jest.advanceTimersByTime()` |
| Compartilhar estado entre testes | Teste B depende de side effect de teste A | `beforeEach` cleanup total |
| `.only` ou `.skip` esquecidos | CI passa mas testes nao rodam | Lint rule `no-only-tests/no-only-tests` |
| Mock granular demais | Testa a implementacao, nao o comportamento | Mock no nivel de HTTP (nock), nao funcoes internas |
| `console.log` para debug | Polui output do CI | `expect(logger.info).toHaveBeenCalledWith(...)` |
| Assercoes vagas (`toBeDefined()`) | Passa mesmo quando valor e errado | `toBe()`, `toEqual()`, `toMatchObject()` |
| Testar mensagem exata de prose | Quebra quando Persona Agent muda fraseado | Testar que funcao foi chamada, nao o conteudo |
