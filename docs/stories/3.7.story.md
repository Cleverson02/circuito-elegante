## DescriГ§ГЈo

Este story implementa o **Golden Discovery #3** вЂ” o Webhook Listener & Auto-Follow-Up вЂ” a capacidade de Stella receber eventos transacionais da Elevare API em tempo real e reagir a eles com mensagens elegantes ao hГіspede, sem polling, sem cron jobs, sem overhead.

**Filosofia:** No luxo, o timing Г© tudo. Quando um link de pagamento estГЎ prestes a expirar, o hГіspede nГЈo deve descobrir por conta prГіpria ao clicar e ver "link invГЎlido". Stella deve antecipar-se вЂ” com a mesma naturalidade com que um concierge humano diria: "Senhor Silva, notei que o link de reserva que preparamos estГЎ prГіximo do prazo. Gostaria que eu providenciasse um novo acesso?" вЂ” discreto, Гєtil, impecГЎvel.

**Problema/Necessidade:** O fluxo de reservas (Stories 3.1-3.3) gera cotaГ§Гµes com links de pagamento que tГЄm validade limitada. Sem o webhook listener, Stella nГЈo sabe quando um link expira, quando um pagamento falha, ou quando uma reserva Г© confirmada. O hГіspede fica desamparado entre a geraГ§ГЈo do link e a conclusГЈo da transaГ§ГЈo вЂ” exatamente o momento mais crГ­tico do funil de conversГЈo.

**DecisГЈo Arquitetural:** O endpoint `POST /webhooks/elevare` recebe eventos da Elevare API com validaГ§ГЈo de assinatura HMAC-SHA256. Cada evento Г© armazenado na tabela `webhook_events` (audit trail imutГЎvel вЂ” APPEND-ONLY, nunca atualizar payload) e processado de forma assГ­ncrona. A idempotГЄncia Г© garantida por hash SHA-256 do payload (`${source}:${eventType}:${JSON.stringify(payload)}`), cacheada em Redis por 24h, evitando processamento duplicado. O processador mapeia cada tipo de evento para dados estruturados que o Persona Agent consome para craftar a mensagem de follow-up no idioma e tom apropriados.

**Fluxo:**
```
Elevare API в†’ POST /webhooks/elevare { event_type, payload, signature }
  в†’ Signature validation (HMAC-SHA256 com ELEVARE_WEBHOOK_SECRET)
  в†’ Idempotency check (SHA-256 hash в†’ Redis dedup cache, 24h TTL)
    в†’ DUPLICATE: 200 OK (acknowledge, don't reprocess)
    в†’ NEW: Insert webhook_events (status: 'received')
  в†’ Process event:
    в†’ Update status: 'received' в†’ 'processing'
    в†’ Lookup guest/session from payload (quotationId в†’ Redis quotation state)
    в†’ Map event_type в†’ follow-up data structure
    в†’ Pass to Persona Agent for luxury prose generation
    в†’ Deliver message via Agent Pipeline
    в†’ Update status: 'processing' в†’ 'processed' (or 'failed' with error_message)
```

## Dev Notes

### Fastify Route Configuration

The webhook endpoint requires access to the **raw body** for HMAC signature validation. Fastify parses JSON bodies by default, but we need the original string to compute the HMAC. Configuration options:

```typescript
// Option 1: Fastify rawBody plugin (recommended)
import fastifyRawBody from 'fastify-raw-body';

await app.register(fastifyRawBody, {
  field: 'rawBody',   // available as request.rawBody
  global: false,       // only on routes that need it
  runFirst: true,      // parse raw body before JSON
  encoding: 'utf8',
});

// Option 2: Custom content type parser
app.addContentTypeParser(
  'application/json',
  { parseAs: 'string' },
  (req, body, done) => {
    try {
      req.rawBody = body;
      done(null, JSON.parse(body as string));
    } catch (err) {
      done(err as Error);
    }
  }
);
```

### HMAC-SHA256 Signature Validation Pattern

```typescript
import { createHmac, timingSafeEqual } from 'node:crypto';

function validateWebhookSignature(
  rawBody: string,
  signature: string,
  secret: string
): boolean {
  const computed = createHmac('sha256', secret)
    .update(rawBody, 'utf8')
    .digest('hex');

  // Timing-safe comparison to prevent timing attacks
  // Both buffers must be the same length for timingSafeEqual
  const sigBuffer = Buffer.from(signature, 'utf8');
  const computedBuffer = Buffer.from(computed, 'utf8');

  if (sigBuffer.length !== computedBuffer.length) {
    return false; // Length mismatch вЂ” invalid signature
  }

  return timingSafeEqual(sigBuffer, computedBuffer);
}
```

**Important:** The `signature` field may come as a header (`x-elevare-signature`) or inside the JSON body. Check Elevare documentation for the exact location. The implementation should support both patterns:
```typescript
const signature = request.headers['x-elevare-signature'] as string
  ?? (request.body as ElevareWebhookBody).signature;
```

### Idempotency Strategy вЂ” SHA-256 Dedup Hash

The dedup strategy uses a SHA-256 hash of `${source}:${eventType}:${JSON.stringify(payload)}` to detect duplicate webhooks. This is cached in Redis with a 24h TTL.

```typescript
import { createHash } from 'node:crypto';

function computeWebhookHash(
  source: string,
  eventType: string,
  payload: unknown
): string {
  const data = `${source}:${eventType}:${JSON.stringify(payload)}`;
  return createHash('sha256').update(data, 'utf8').digest('hex');
}

// Redis key: webhook_dedup:{hash}
// Value: "1" (existence check only)
// TTL: 24h
```

**Why not use Elevare's `eventId`?** If Elevare provides an `eventId`, prefer that for dedup (simpler, more reliable). The hash-based approach is the fallback when no unique event ID is provided by the upstream. The implementation should check for `eventId` first:
```typescript
const dedupKey = payload.eventId
  ? `webhook_dedup:${payload.eventId}`
  : `webhook_dedup:${computeWebhookHash('elevare', eventType, payload)}`;
```

### webhook_events Drizzle Insert

```typescript
import { db } from '../database/client.js';
import { webhookEvents } from '../database/schema.js';

async function insertWebhookEvent(data: {
  source: string;
  eventType: string;
  payload: Record<string, unknown>;
  sessionId: string | null;
  guestId: string | null;
}): Promise<string> {
  const [result] = await db
    .insert(webhookEvents)
    .values({
      source: data.source,
      eventType: data.eventType,
      payload: data.payload,
      sessionId: data.sessionId,
      guestId: data.guestId,
      status: 'received',
    })
    .returning({ id: webhookEvents.id });

  return result.id;
}

// Status update вЂ” NEVER touch payload column
async function updateWebhookEventStatus(
  id: string,
  status: 'processing' | 'processed' | 'failed',
  errorMessage?: string
): Promise<void> {
  await db
    .update(webhookEvents)
    .set({
      status,
      processedAt: new Date(),
      ...(errorMessage && { errorMessage }),
    })
    .where(eq(webhookEvents.id, id));
}
```

### Guest Resolution Flow

```typescript
async function resolveGuestContext(
  quotationId: string
): Promise<GuestContext | null> {
  // Step 1: Check Redis for quotation state (fast path)
  const quotationState = await cacheGet<QuotationState>(
    REDIS_KEYS.quotation(quotationId)
  );

  if (quotationState) {
    return {
      sessionId: quotationState.sessionId,
      guestId: quotationState.customerId,
      guestName: undefined, // Resolve from session or guest_profiles if needed
      language: 'pt',       // Default, resolve from session
    };
  }

  // Step 2: Fallback to DB вЂ” search historical webhook_events
  // for any event with this quotationId in payload
  const historicalEvent = await db
    .select()
    .from(webhookEvents)
    .where(
      sql`payload->>'quotationId' = ${quotationId}`
    )
    .orderBy(desc(webhookEvents.createdAt))
    .limit(1);

  if (historicalEvent.length > 0 && historicalEvent[0].sessionId) {
    return {
      sessionId: historicalEvent[0].sessionId,
      guestId: historicalEvent[0].guestId,
      guestName: undefined,
      language: 'pt',
    };
  }

  // Step 3: Cannot resolve guest вЂ” event will be stored but no follow-up
  return null;
}
```

### Status State Machine

```
received в”Ђв”Ђв†’ processing в”Ђв”Ђв†’ processed
                 в”‚
                 в””в”Ђв”Ђв†’ failed (with error_message)
```

- `received`: Webhook accepted, stored in DB, not yet processed
- `processing`: Event handler executing (guest lookup, follow-up generation)
- `processed`: Follow-up sent successfully (or event is audit-only like `quote_created`)
- `failed`: Processing error (guest not resolved, Persona Agent failure, delivery failure)

**Transitions are ONE-WAY.** A `failed` event is never retried automatically in this story (future: retry queue). Manual replay is possible via admin tools (future story).

### Error Recovery

| Error Scenario | Handler Behavior | HTTP Response |
|----------------|-----------------|---------------|
| Invalid HMAC signature | Reject immediately | 401 Unauthorized |
| Missing ELEVARE_WEBHOOK_SECRET | Reject immediately | 503 Service Unavailable |
| Malformed body (missing event_type) | Reject immediately | 400 Bad Request |
| Duplicate webhook (dedup hash exists) | Acknowledge, skip processing | 200 OK `{ status: 'duplicate' }` |
| Unknown event_type | Store in DB, skip processing | 200 OK `{ status: 'received' }` |
| Guest not resolved | Store as processed with error_message | 200 OK `{ status: 'processed' }` |
| Persona Agent failure | Store as failed with error_message | 200 OK `{ status: 'failed' }` |
| DB insert failure | Log critical error | 500 Internal Server Error |
| Redis unavailable | Process without dedup (log warning) | 200 OK |

**Critical principle:** Always return 200 OK to Elevare after signature validation passes. If we return 4xx/5xx for processing errors, Elevare may retry indefinitely, creating a feedback loop. The only non-200 responses are security rejections (401, 503) and infrastructure failures (500 from DB).

### Rate Limiting Implementation

```typescript
// Option 1: Fastify rate-limit plugin
import rateLimit from '@fastify/rate-limit';

app.register(rateLimit, {
  max: 100,
  timeWindow: '1 minute',
  keyGenerator: (request) => request.ip,
  // Only apply to webhook routes
  allowList: [], // No exemptions
});

// Option 2: Custom Redis counter (if more control needed)
async function checkRateLimit(ip: string): Promise<boolean> {
  const key = `rate_limit:webhook:${ip}`;
  const count = await redis.incr(key);
  if (count === 1) {
    await redis.expire(key, 60); // 1 minute window
  }
  return count <= 100;
}
```
