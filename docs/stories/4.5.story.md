## Descricao

Este story implementa a **correfencia via WhatsApp Reply** — o mecanismo que permite ao hospede tocar em uma imagem especifica de hotel (enviada pela renderizacao premium, Story 4.4) e responder diretamente via Reply do WhatsApp, e Stella resolve automaticamente para o `offerId` correto sem pedir confirmacao adicional.

**Problema/Necessidade:** Apos a curadoria (Story 3.4) e a renderizacao premium (Story 4.4) enviarem 3-4 opcoes com foto + legenda nativa no WhatsApp, o hospede tem duas formas de indicar sua escolha: (1) texto livre com ordinal/superlativo ("a segunda", "a mais cara") — ja implementado na Story 3.5, ou (2) **Reply diretamente na imagem** — gesto natural e intuitivo no WhatsApp. A opcao (2) e este story.

**Por que WhatsApp Reply e essencial:**

Em experiencias de luxury concierge, o hospede nao quer digitar — quer apontar. No WhatsApp, o equivalente a "apontar para a foto" e o gesto de Reply (swipe + resposta). E o caminho de menor friccao: o hospede ve as fotos, desliza na que gostou, e digita "quero essa" ou simplesmente "sim". Stella precisa saber QUAL foto foi referenciada.

**Arquitetura — Dois Lados da Moeda:**

O mecanismo de correfencia via Reply tem duas partes independentes que este story conecta:

1. **Lado de Saida (outgoing):** Quando o typing-worker (Story 4.3) envia imagens via Evolution API, a API retorna um `messageId` unico para cada mensagem enviada (`EvolutionResponse.key.id`). Hoje, o typing-worker **ignora** esse retorno. Story 4.5 captura esses IDs e os registra no OfferMap via `updateOfferMapMessageIds()` (Story 3.5).

2. **Lado de Entrada (incoming):** Quando o hospede faz Reply a uma mensagem, o webhook da Evolution API inclui o campo `contextInfo.stanzaId` com o `messageId` da mensagem original. O webhook handler (Story 4.1) ja extrai isso como `ParsedMessage.quotedMessageId`. Story 4.5 propaga esse campo ate o `resolveCoreference()` (Story 3.5) que ja sabe resolver por `whatsappMessageId`.

**O Gap que Story 4.5 Preenche:**

```
HOJE (gap):
  typing-worker envia imagem → ignora EvolutionResponse.key.id ← CAPTURAR
  webhook recebe Reply → ParsedMessage.quotedMessageId existe ← PROPAGAR
  buffer consolida texto → quotedMessageId se perde ← PRESERVAR
  buffer-processor → nao chama resolveCoreference() ← INTEGRAR

DEPOIS (Story 4.5):
  typing-worker envia imagem → captura key.id → updateOfferMapMessageIds()
  webhook recebe Reply → ParsedMessage.quotedMessageId → buffer preserva
  buffer-processor detecta quotedMessageId → resolveCoreference({ whatsappMessageId })
  → offerId resolvido → pipeline prossegue com selecao
```

## Acceptance Criteria

- [x] AC1: `TypingChunk` e `TypingJobData` types (em `backend/src/queue/types.ts`) estendidos com campo opcional `curatedPosition?: number` indicando a posicao (1-indexed) da opcao curada que a imagem representa. `enqueueResponse()` em `typing-queue.ts` copia `curatedPosition` do chunk para o job data. Chunks/jobs sem curatedPosition permanecem inalterados.
- [x] AC2: `renderCuratedOptions()` (em `backend/src/services/media-renderer.ts`) define `curatedPosition` no chunk de imagem de cada opcao — posicao baseada no `curatedRank` da `CuratedOption`. Apenas chunks de imagem (`isMedia: true`) recebem `curatedPosition`; chunks de legenda nao.
- [x] AC3: `typing-worker.ts` captura o retorno `EvolutionResponse.key.id` ao enviar chunks com `curatedPosition`. Apos envio bem-sucedido, chama `updateOfferMapMessageIds(sessionId, { [curatedPosition]: messageId })` para registrar o mapeamento no OfferMap Redis.
- [x] AC4: Se `EvolutionResponse` nao retorna `key.id` (erro, timeout, ou resposta incompleta), o worker loga warning `media_message_id_capture_failed` com `{ sessionId, phone, curatedPosition, chunkIndex }` e **continua normalmente** — a opcao fica sem `whatsappMessageId` e resolve apenas por texto/ordinal.
- [x] AC5: `message-buffer.ts` preserva `quotedMessageId` durante a consolidacao de mensagens. Quando qualquer mensagem no batch de 20s possui `quotedMessageId`, o campo e propagado ao callback do buffer-processor. Se multiplas mensagens possuem `quotedMessageId`, manter o mais recente (ultimo timestamp).
- [x] AC6: Callback do buffer-processor recebe `quotedMessageId` opcional. Quando presente, antes de chamar o pipeline de agentes, tenta resolver via `resolveCoreference(sessionId, { whatsappMessageId: quotedMessageId })`.
- [x] AC7: Se `resolveCoreference()` retorna `{ resolved: true, option }`, o buffer-processor enriquece o input do pipeline com `{ resolvedOfferId: option.offerId, resolvedOption: option, selectionMethod: 'whatsapp_reply' }` para que o orchestrator prossiga diretamente para quotation sem re-curadoria.
- [x] AC8: Se `resolveCoreference()` retorna `{ resolved: false }` (messageId nao encontrado, mapa expirado, ou clarificacao necessaria), o buffer-processor segue o fluxo normal — passa a mensagem consolidada ao pipeline para processamento via texto/ordinal (graceful fallback).
- [x] AC9: Log estruturado para cada resolucao por Reply: `{ event: 'coreference_reply_resolved', sessionId, quotedMessageId, resolvedOfferId, curatedPosition }` quando resolvido, ou `{ event: 'coreference_reply_fallback', sessionId, quotedMessageId, reason }` quando fallback.
- [x] AC10: Testes unitarios cobrindo: (a) captura de messageId no typing-worker, (b) updateOfferMapMessageIds chamado com posicao correta, (c) quotedMessageId preservado pelo buffer, (d) resolucao bem-sucedida via Reply, (e) fallback quando messageId nao encontrado, (f) fallback quando EvolutionResponse sem key.id, (g) multiplas mensagens com quotedMessageId — cobertura >= 90% branches.
- [x] AC11: Teste de integracao round-trip: enviar 3 opcoes curadas → capturar messageIds → simular Reply a imagem 2 → verificar que offerId da opcao 2 e resolvido corretamente.

### Fluxo Completo (Apos Story 4.5)

```
1. Curadoria (3.4) → CuratedOption[] com photos
2. createOfferMap() (3.5) → Redis offers:{sessionId} (posicoes 1-4, sem messageIds)
3. renderCuratedOptions() (4.4) → TypingChunk[] com curatedPosition nos chunks de imagem
4. enqueueResponse() → BullMQ queue
5. typing-worker processa cada chunk:
   - Imagem com curatedPosition → sendText() → captura key.id → updateOfferMapMessageIds()
   - Legenda → sendText() com typing delay
6. OfferMap atualizado no Redis: posicoes 1-4 com whatsappMessageIds

[... hospede ve as fotos, desliza na que gostou ...]

7. Hospede faz Reply na imagem 2 → webhook recebe com quotedMessageId
8. message-buffer consolida (20s) → preserva quotedMessageId
9. buffer-processor detecta quotedMessageId → resolveCoreference(sessionId, { whatsappMessageId })
10. resolveByMessageId() → OfferOption posicao 2 → offerId
11. Pipeline recebe resolvedOfferId → prossegue para quotation (3.3)
```

### Technical Constraints
- **updateOfferMapMessageIds e read-modify-write:** Cada chamada le o OfferMap do Redis, atualiza uma posicao, e reescreve. Como o typing-worker processa chunks sequencialmente (com delays), nao ha race condition pratica. [Source: coreference.ts:311-345]
- **Buffer consolida texto mas precisa preservar metadata:** A modificacao no buffer deve ser minima — adicionar tracking de um campo opcional, nao reestruturar o buffer inteiro. [Source: message-buffer.ts]
- **Fallback e king:** Se qualquer parte do mecanismo de Reply falhar, o hospede ainda pode usar texto ("a segunda") e o OrdinalParser resolve normalmente. [Source: coreference.ts resolveCoreference()]
