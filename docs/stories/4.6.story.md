## Descricao

Este story implementa o **chat widget basico para website** — o segundo canal de comunicacao da Stella alem do WhatsApp. Um widget embed HTML/JS que conecta via WebSocket ao backend Fastify e permite que visitantes do site conversem com Stella em tempo real.

**Problema/Necessidade:** O WhatsApp e o canal primario, mas muitos hospedes descobrem hoteis pelo website primeiro. Um chat widget no canto inferior do site permite que o visitante converse com Stella sem sair da pagina — sem instalar app, sem compartilhar numero de telefone. E a porta de entrada para conversoes de website.

**Diferenca fundamental Website vs WhatsApp:**
- **WhatsApp:** Buffer 20s (mensagens picadas), typing simulation, imagens nativas, reply coreference
- **Website:** Resposta imediata (sem buffer), rendering em HTML/CSS, sem typing events nativos (simulacao visual), imagens via `<img>` tags

**Arquitetura:**

O widget e um arquivo HTML/JS estatico que abre WebSocket com `ws://backend/ws/chat`. O backend usa `@fastify/websocket` para aceitar conexoes. Cada conexao WebSocket recebe um `sessionId` e as mensagens fluem pelo mesmo pipeline de agentes que o WhatsApp — apenas a camada de entrega muda.

O `typing-worker.ts` ja tem um placeholder para website (linhas 41-52) que loga `typing_chunk_website` mas nao envia. Story 4.6 conecta esse placeholder ao WebSocket.

```
Website Widget → WebSocket → Fastify → Agent Pipeline → Response
                                                          ↓
                                    typing-worker (channel='website')
                                                          ↓
                                    WebSocket.send(sessionId, { type: 'message', text })
```

- [x] AC8: Typing-worker (linhas 41-52) modificado: em vez de placeholder log, envia mensagem via `WebSocketManager.send(sessionId, { type: 'message', text })`. Se conexao nao existe (cliente desconectou), log warning e descarta chunk.

## Dev Notes

### Previous Story Insights
- **Story 4.3 (Typing Simulation):** typing-worker.ts linhas 41-52 tem placeholder website channel que loga mas nao envia. Story 4.6 conecta esse placeholder ao WebSocket real. [Source: typing-worker.ts]
- **Story 4.2 (Buffer):** MessageBuffer NAO deve ser usado para website — website e resposta imediata. [Source: message-buffer.ts]
- **Story 4.4 (Premium Rendering):** Para channel 'website', `renderCuratedOptions()` gera chunks de texto (sem imagem nativa). Widget futuramente renderizara imagens via HTML. [Source: media-renderer.ts AC11]

### Data Models
- **WebSocket Message (client → server):** `{ type: 'message', text: string }`
- **WebSocket Message (server → client):** `{ type: 'message', text: string, chunkIndex: number, totalChunks: number }`
- **WebSocket Welcome:** `{ type: 'welcome', sessionId: string, message: string }`
- **WebSocket Error:** `{ type: 'error', message: string }`

### Technical Constraints
- **@fastify/websocket** e o plugin oficial — nao usar socket.io (over-engineered para MVP) [Source: architecture.md]
- **Sessao anonima:** sessionId = UUID v4, sem autenticacao. Sessao vive enquanto WebSocket esta aberto.
- **Single-instance:** WebSocketManager em memoria (Map). Para multi-instance futuro, precisaria Redis pub/sub — fora do escopo MVP (NFR8: 5000 msgs/month).
- **CORS:** Widget standalone pode ser servido de qualquer dominio. Backend deve aceitar origin do widget. [Source: server.ts middleware]
