## Descrição

Este story implementa a **consulta de status de reservas pós-venda** — a capacidade de Stella responder quando um hóspede retorna após ter completado uma reserva e pergunta "Como está minha reserva?", "Quero saber o status do meu booking" ou simplesmente envia um código de confirmação.

**Filosofia:** O pós-venda em hotelaria de luxo é onde se consolida a confiança. Quando um hóspede já confiou seu cartão, seus dados pessoais e suas datas de viagem a Stella, a última coisa que ele quer é ouvir "não consigo verificar, ligue para a recepção". Stella deve ser tão competente no acompanhamento quanto foi na venda. A resposta deve ser imediata, precisa e — acima de tudo — humana: datas formatadas no idioma do hóspede, sem códigos técnicos, sem jargão de sistema, sem IDs opacos. Apenas informação clara: "Sua reserva no Hotel Fasano está confirmada para 15 de março. Esperamos você!"

**Problema/Necessidade:** Após o fluxo de ouro (search → customer → quotation → payment), o hóspede perde visibilidade sobre sua reserva. Atualmente, para verificar status, ele precisa ligar para a central ou enviar email. Com `check_reservation`, Stella resolve isso em segundos via WhatsApp — usando a mesma API Elevare que originou a reserva.

**Decisão Arquitetural:** O `check_reservation` é uma **tool do Orchestrator** — segue o mesmo pattern de `search_hotels` e `transfer_to_human`. A tool query a API Elevare (`GET /global-agent/reservations`) usando um ou mais identificadores do hóspede (código de confirmação, email ou telefone). O resultado bruto da API é mapeado para um `ReservationStatusData` — estrutura enxuta com dados humanizáveis que o Persona Agent consome para gerar a resposta final no idioma da sessão. Nenhum ID técnico, nenhum campo interno, nenhum jargão de API chega ao hóspede.

**Fluxo:**
```
Hóspede: "Qual o status da minha reserva ABC123?"
  → Intent Agent classifica: STATUS
  → Orchestrator recebe intent STATUS
  → Orchestrator chama check_reservation(identifier: "ABC123")
  → Tool identifica tipo: confirmationNumber
  → ElevareClient.getReservations({ confirmationNumber: "ABC123" })
  → GET /global-agent/reservations?confirmationNumber=ABC123
  → Elevare retorna: { reservations: [...] }
  → Tool mapeia para ReservationStatusData (human-readable)
  → Orchestrator passa dados brutos ao Persona Agent
  → Persona Agent gera: "Sua reserva no Hotel Fasano está confirmada!
     Check-in: sábado, 15 de março de 2026. Esperamos você!"
```

## Acceptance Criteria

- [ ] AC1: Tool `check_reservation` registrada no Orchestrator Agent com `name: 'check_reservation'` — segue pattern de `search_hotels` e `transfer_to_human` (OpenAI Agents SDK `tool()`)
- [ ] AC2: Tool aceita parâmetro `identifier` (string obrigatória) que pode ser: código de confirmação (e.g., `"ABC123"`), email (e.g., `"joao@email.com"`), ou telefone (e.g., `"+5511999887766"`) — a tool detecta automaticamente o tipo
- [ ] AC3: Detecção de tipo do identificador: email validado por regex (`/@/`), telefone validado por regex (`/^\+?\d{10,15}$/`), qualquer outro valor tratado como `confirmationNumber` — sem ambiguidade
- [ ] AC4: Query à API Elevare via `GET /global-agent/reservations` usando o query parameter correspondente ao tipo detectado: `confirmationNumber`, `email`, ou `guestPhone` — reutiliza `ElevareClient` de Story 3.1 (retry, circuit breaker, timeout, auth)
- [ ] AC5: Autenticação via headers `x-client-id` e `x-client-secret` (mesmo padrão da Postman collection Elevare Global Agent) — credenciais lidas de env vars `ELEVARE_CLIENT_ID` e `ELEVARE_CLIENT_SECRET`, nunca hardcoded
- [ ] AC6: Resposta da API mapeada para `ReservationStatusData` com campos humanizáveis: `hotelName`, `guestName`, `checkIn` (ISO date), `checkOut` (ISO date), `status` (enum mapeado), `roomType`, `totalPrice`, `currency`, `confirmationNumber` — IDs técnicos internos (`hotelId`, `customerId`, `quotationId`) são DESCARTADOS no mapeamento
- [ ] AC7: Status da API mapeado para enum interno: `confirmed`, `pending_payment`, `cancelled`, `checked_in`, `checked_out` — status desconhecidos mapeados para `unknown` com logging de warning
- [ ] AC8: Quando múltiplas reservas são encontradas (e.g., busca por email retorna 3 reservas), a tool retorna TODAS como array de `ReservationStatusData` — Persona Agent decide como apresentar (lista ou pergunta qual)
- [ ] AC9: Quando nenhuma reserva é encontrada, a tool retorna `{ found: false, suggestion: 'front_desk' }` — Persona Agent gera fallback elegante ("Não encontrei reservas com esse dado. Posso conectá-lo com nossa recepção?")
- [ ] AC10: `ReservationStatusData` inclui dados para formatação localizada: `checkIn` e `checkOut` como strings ISO 8601 (`YYYY-MM-DD`) para que o Persona Agent formate com `Intl.DateTimeFormat` no idioma da sessão
- [ ] AC11: Nenhum ID técnico chega ao Persona Agent: campos como `hotelId`, `customerId`, `requestId`, `quotationId`, `offerId` são filtrados no mapeamento — apenas dados humanizáveis passam
- [ ] AC12: A tool retorna dados estruturados para o Persona Agent gerar mensagens status-aware com mapeamento semântico:
  - `confirmed` → dados para: "Reserva confirmada, esperamos em {checkIn}"
  - `pending_payment` → dados para: "Aguardando pagamento"
  - `cancelled` → dados para: "Reserva cancelada, deseja nova consulta?"
  - `checked_in` → dados para: "Check-in realizado, bom descanso!"
  - `checked_out` → dados para: "Check-out realizado, obrigada pela visita!"
- [ ] AC13: Quando a API Elevare falha (timeout, 5xx, circuit breaker aberto), a tool retorna `{ found: false, error: true, suggestion: 'front_desk' }` — Persona Agent gera fallback sem expor erro técnico ("No momento não consigo acessar o sistema de reservas. Posso transferi-lo para a recepção?")
- [ ] AC14: Logging estruturado em toda operação: `logger.info('reservation_lookup', { identifierType, resultCount })` no sucesso; `logger.warn('reservation_not_found', { identifierType })` quando vazio; `logger.error('reservation_lookup_failed', { identifierType, error })` em falha — NUNCA logar o identificador completo (PII: email, telefone)
- [ ] AC15: Tipos TypeScript exportados: `CheckReservationParams`, `ReservationStatusData`, `ReservationStatus` (enum), `ReservationLookupResult` — todos em arquivo de types separado

## Dev Notes

### Elevare Reservations API — Endpoint Detalhado

**Fonte:** `docs/api/elevare-global-agent.postman_collection.json` — "Search Global Reservations"

```
GET /global-agent/reservations
  Headers:
    x-client-id: {ELEVARE_CLIENT_ID}
    x-client-secret: {ELEVARE_CLIENT_SECRET}
  Query Params (todos opcionais, pelo menos 1 obrigatório):
    - confirmationNumber: string  (e.g., "ABC123")
    - email: string               (e.g., "joao.silva@email.com")
    - guestName: string            (e.g., "João Silva")
    - hotelId: string              (e.g., "HOTEL_CODE")
    - checkInFrom: string          (e.g., "2025-01-01")
    - checkInTo: string            (e.g., "2025-12-31")
    - limit: number                (e.g., 10)
```

**IMPORTANTE — Diferença de autenticação:** A Postman collection mostra que o endpoint `/global-agent/reservations` usa `x-client-id` + `x-client-secret` (client credentials), enquanto os endpoints de search usam `X-Api-Key`. Isto pode significar que o ElevareClient de Story 3.1 precisa suportar ambos auth patterns, ou que o módulo de reservations faz seu próprio fetch com auth diferente. A implementação deve verificar qual auth pattern funciona e implementar adequadamente.

**Nota sobre phone:** O Postman collection não mostra `phone` como query param explícito para `/reservations`. O parâmetro disponível mais próximo é `guestName`. Para lookup por telefone, pode ser necessário usar o endpoint `GET /global-agent/customers/phone/{phone}` primeiro para obter o `customerId`, depois buscar reservas associadas. **Decisão de implementação:** Tentar `email` e `confirmationNumber` primeiro (confirmados no Postman). Para telefone, implementar como best-effort (tentar `guestName` fallback ou customer lookup) e se não funcionar, retornar `{ found: false, suggestion: 'front_desk' }`.

### Tool Registration Pattern

Seguir o pattern exato dos tools existentes em `backend/src/tools/`:

```typescript
// backend/src/tools/check-reservation.ts
import { z } from 'zod';
import { tool } from '@openai/agents';
import { getReservations } from '../integrations/elevare/reservations.js';

export const CheckReservationParams = z.object({
  identifier: z.string().describe(
    'Guest identifier to lookup reservation: confirmation code (e.g., "ABC123"), ' +
    'email (e.g., "guest@email.com"), or phone (e.g., "+5511999887766")'
  ),
});

export type CheckReservationParams = z.infer<typeof CheckReservationParams>;

export const checkReservationTool = tool({
  name: 'check_reservation',
  description:
    'Check the status of an existing reservation. Guest can provide their ' +
    'confirmation code, email address, or phone number. Returns structured ' +
    'reservation data (hotel name, dates, status, room type) for Persona Agent ' +
    'to format as a human-readable response. Use when intent is STATUS.',
  parameters: CheckReservationParams,
  execute: async (params) => {
    const result = await getReservations(params.identifier);
    if (!result.found) {
      return {
        found: false,
        error: result.error ?? false,
        suggestion: 'front_desk',
        message: 'No reservations found for this identifier.',
      };
    }
    return {
      found: true,
      reservations: result.reservations,
      count: result.reservations.length,
    };
  },
});
```

### Identifier Detection — Implementation

```typescript
interface IdentifierInfo {
  type: 'confirmationNumber' | 'email' | 'phone';
  queryParam: string;  // Elevare API query parameter name
  value: string;       // Normalized value
}

function detectIdentifierType(identifier: string): IdentifierInfo {
  const trimmed = identifier.trim();

  // Email: contains @ with basic format
  if (/@/.test(trimmed) && /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(trimmed)) {
    return {
      type: 'email',
      queryParam: 'email',
      value: trimmed.toLowerCase(),
    };
  }

  // Phone: starts with + or has 10-15 digits (strip spaces/hyphens first)
  const phoneNormalized = trimmed.replace(/[\s\-()]/g, '');
  if (/^\+?\d{10,15}$/.test(phoneNormalized)) {
    return {
      type: 'phone',
      queryParam: 'email', // NOTE: Elevare may not support phone directly
      // Fallback strategy: try customer lookup by phone first
      value: phoneNormalized.startsWith('+') ? phoneNormalized : `+${phoneNormalized}`,
    };
  }

  // Default: confirmation number
  return {
    type: 'confirmationNumber',
    queryParam: 'confirmationNumber',
    value: trimmed.toUpperCase(),
  };
}
```

### Response Mapping — Elevare → ReservationStatusData

A resposta da API Elevare é mapeada para dados humanizáveis, descartando IDs técnicos:

```typescript
// Mapeamento esperado (inferido da Postman collection e arquitetura)
interface ElevareReservationResponse {
  reservations: Array<{
    // IDs técnicos — DESCARTAR
    id: string;
    hotelId: string;
    customerId: string;
    quotationId?: string;

    // Dados humanizáveis — MANTER
    confirmationNumber: string;
    hotelName: string;
    guestName: string;
    checkIn: string;           // "2026-03-15"
    checkOut: string;          // "2026-03-18"
    status: string;            // Elevare status string
    roomType: string;          // "Suite Master"
    totalPrice: number;        // 2500.00
    currency: string;          // "BRL"
  }>;
}

function mapToStatusData(raw: ElevareReservationResponse['reservations'][0]): ReservationStatusData {
  return {
    hotelName: raw.hotelName ?? 'Hotel',
    guestName: raw.guestName ?? '',
    checkIn: raw.checkIn,
    checkOut: raw.checkOut,
    status: mapStatus(raw.status),
    roomType: raw.roomType ?? '',
    totalPrice: raw.totalPrice ?? 0,
    currency: raw.currency ?? 'BRL',
    confirmationNumber: raw.confirmationNumber ?? '',
  };
}
```

### Status Mapping Table

| Elevare API Status | Internal Enum | Persona Agent Semantic |
|-------------------|---------------|----------------------|
| `"confirmed"` | `confirmed` | Reserva confirmada, esperamos em {checkIn} |
| `"pending"` / `"pending_payment"` / `"awaiting_payment"` | `pending_payment` | Aguardando pagamento |
| `"cancelled"` / `"canceled"` | `cancelled` | Reserva cancelada, gostaria de fazer nova consulta? |
| `"checked_in"` / `"in_house"` | `checked_in` | Check-in realizado, tenha uma excelente estadia! |
| `"checked_out"` / `"departed"` | `checked_out` | Check-out realizado, obrigada pela confiança! |
| Qualquer outro valor | `unknown` | Log warning + fallback genérico |

**Nota:** A Elevare API pode usar nomenclaturas diferentes das listadas acima. O mapeamento é defensivo — aceita variações comuns e loga warnings para valores desconhecidos. Se durante a integração real os status forem diferentes, basta atualizar a lookup table sem alterar a interface.

```typescript
const STATUS_MAP: Record<string, ReservationStatus> = {
  'confirmed': 'confirmed',
  'active': 'confirmed',
  'pending': 'pending_payment',
  'pending_payment': 'pending_payment',
  'awaiting_payment': 'pending_payment',
  'cancelled': 'cancelled',
  'canceled': 'cancelled',
  'checked_in': 'checked_in',
  'in_house': 'checked_in',
  'checked_out': 'checked_out',
  'departed': 'checked_out',
  'completed': 'checked_out',
};

function mapStatus(elevareStatus: string): ReservationStatus {
  const normalized = elevareStatus.toLowerCase().trim();
  const mapped = STATUS_MAP[normalized];
  if (!mapped) {
    logger.warn('unknown_reservation_status', { elevareStatus });
    return 'unknown';
  }
  return mapped;
}
```

### Persona Agent Data Structure — What Gets Passed

O Persona Agent recebe `ReservationStatusData` e gera prosa adequada ao idioma da sessão. Exemplos de como o Persona Agent consome os dados:

**Status: confirmed (PT-BR)**

### PII Protection in Logs

```typescript
function maskPII(identifier: string, type: string): string {
  switch (type) {
    case 'email': {
      const [local, domain] = identifier.split('@');
      return `${local[0]}***@${domain}`;
    }
    case 'phone': {
      return `${identifier.slice(0, 4)}***${identifier.slice(-4)}`;
    }
    default:
      return identifier; // Confirmation codes are not PII
  }
}

// Usage in logging:
logger.info('reservation_lookup', {
  identifierType: info.type,
  identifierMasked: maskPII(info.value, info.type),
  resultCount: reservations.length,
});
```

### Auth Pattern — x-client-id/secret vs X-Api-Key

A Postman collection (`elevare-global-agent.postman_collection.json`) mostra dois patterns de autenticação:

1. **Search endpoints** (`/search`, `/multi-search`): `X-Api-Key` header → `ELEVARE_API_KEY` env var
2. **Global Agent endpoints** (`/global-agent/reservations`, `/global-agent/customers`): `x-client-id` + `x-client-secret` headers → `ELEVARE_CLIENT_ID` + `ELEVARE_CLIENT_SECRET` env vars

O módulo `reservations.ts` deve usar o padrão (2). Se o `ElevareClient` de Story 3.1 não suportar esse auth pattern, criar um método ou wrapper dedicado que use as credenciais corretas. Adicionar as env vars `ELEVARE_CLIENT_ID` e `ELEVARE_CLIENT_SECRET` ao Zod schema de configuração em `config/env.ts`.
