## DescriГ§ГЈo

Criar o `ElevareClient` вЂ” o cliente TypeScript production-grade para a API Elevare вЂ” com capacidade de busca de disponibilidade (`GET /search`), multi-search por cidade/regiГЈo (`GET /multi-search`), e gerenciamento de `requestId` em Redis. Este Г© o alicerce de todo o "Fluxo de Ouro" (search в†’ customer в†’ quotation в†’ payment link) e serГЎ estendido nas stories 3.2 e 3.3 com endpoints adicionais.

**Problema/Necessidade:** O Orchestrator Agent precisa consultar disponibilidade de quartos em tempo real para os 32 hotГ©is integrados via API Elevare. Sem o ElevareClient, Stella Г© apenas um chatbot informativo вЂ” com ele, torna-se um agente de reservas funcional. O client deve ser resiliente (circuit breaker, retry com jitter, timeout 8s), observГЎvel (logging estruturado com Winston), e seguro (API key nunca exposta em logs ou ao hГіspede, requestId opaco em Redis).

**Golden Discovery #1:** A API Elevare retorna FOTOS dos quartos no payload do `/search`. NГЈo Г© necessГЎrio armazenar imagens no PostgreSQL вЂ” elas vГЄm diretamente da resposta transacional. Isso simplifica drasticamente a renderizaГ§ГЈo de assets (FR22).

**Golden Discovery #2:** O fluxo Elevare Г© sequencial e baseado em tokens opacos: `search` retorna `requestId` в†’ `customer` retorna `customerId` в†’ `quotation` usa ambos para gerar `paymentLink`. O `requestId` tem vida Гєtil limitada (TTL 30min no Redis) e NUNCA deve ser exposto ao hГіspede via WhatsApp (FR24).

### Circuit Breaker вЂ” State Machine

```
              5 consecutive failures
  CLOSED в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв–є OPEN
    в–І                                    в”‚
    в”‚ 3 consecutive                      в”‚ 60s cooldown
    в”‚ successes in                       в”‚ elapsed
    в”‚ half-open                          в–ј
    в””в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ HALF_OPEN
                                    в”‚
                                    в”‚ failure in half-open
                                    в–ј
                                   OPEN (reset cooldown)
```

**ImplementaГ§ГЈo:**

```typescript
interface CircuitBreakerState {
  status: 'CLOSED' | 'OPEN' | 'HALF_OPEN';
  failureCount: number;           // Consecutivos em CLOSED
  successCount: number;           // Consecutivos em HALF_OPEN
  lastFailureAt: number | null;   // timestamp
  nextAttemptAt: number | null;   // timestamp para transiГ§ГЈo OPEN в†’ HALF_OPEN
}

const CIRCUIT_BREAKER_CONFIG = {
  failureThreshold: 5,      // Falhas para abrir
  successThreshold: 3,      // Sucessos em half-open para fechar
  cooldownMs: 60_000,       // 60s em OPEN antes de HALF_OPEN
};
```

**Regras:**
- Em CLOSED: cada falha incrementa `failureCount`. Sucesso reseta para 0.
- Ao atingir 5 falhas: transiciona para OPEN, seta `nextAttemptAt = now + 60s`.
- Em OPEN: rejeita imediatamente com `ElevareCircuitOpenError` (fail-fast).
- ApГіs 60s: transiciona para HALF_OPEN, permite 1 request de teste.
- Em HALF_OPEN: sucesso incrementa `successCount`. 3 sucessos в†’ CLOSED. Qualquer falha в†’ OPEN (reset cooldown).

### Retry com Jitter вЂ” ImplementaГ§ГЈo

```typescript
function calculateRetryDelay(attempt: number, baseMs: number = 500): number {
  // Exponential backoff: 500ms, 1000ms, 2000ms
  const exponentialDelay = baseMs * Math.pow(2, attempt);
  // Jitter: В±30% randomizado
  const jitterFactor = 1 + (Math.random() * 0.6 - 0.3); // 0.7 a 1.3
  return Math.round(exponentialDelay * jitterFactor);
}
// attempt 0: 350ms - 650ms
// attempt 1: 700ms - 1300ms
// attempt 2: 1400ms - 2600ms
```

**Regras de retry:**
- Retry APENAS em: 5xx (server error), `ECONNRESET`, `ECONNREFUSED`, `ETIMEDOUT`, `AbortError` (timeout)
- NUNCA retry em: 4xx (client error вЂ” 400, 401, 403, 404, 422)
- Max 3 tentativas (configurГЎvel via `ELEVARE_MAX_RETRIES`)

### Timeout вЂ” AbortController Pattern

```typescript
async function fetchWithTimeout(url: string, options: RequestInit, timeoutMs: number): Promise<Response> {
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), timeoutMs);
  try {
    const response = await fetch(url, { ...options, signal: controller.signal });
    return response;
  } catch (error) {
    if (error instanceof DOMException && error.name === 'AbortError') {
      throw new ElevareTimeoutError(timeoutMs, url);
    }
    throw error;
  } finally {
    clearTimeout(timeoutId);
  }
}
```

**Timeout:** 8000ms default (configurГЎvel via `ELEVARE_TIMEOUT_MS`). Alinhado com NFR1 (pipeline total < 10s, Orchestrator < 3s + margem).
