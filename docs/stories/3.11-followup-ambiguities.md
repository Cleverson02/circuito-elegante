# Story 3.11 — Followup: Ambiguidades Pendentes da Elevare

**Contexto:** a Story 3.11 corrigiu auth, paths e env schema com base no que a Postman collection (`docs/api/elevare-global-agent.postman_collection.json`) deixa explicito. Os pontos abaixo NAO estao documentados de forma univoca na Postman — precisam de confirmacao da Elevare antes de virarem codigo.

Quando todos (ou alguns) forem respondidos, estes itens viram **Story 3.12**.

---

## 1. Schema do webhook inbound (Elevare → Stella)

**Status no codigo hoje:** `backend/src/webhooks/elevare-handler.ts` processa os event types `quote_expiring`, `reservation_confirmed`, `payment_failed`, `quote_created`. Sinatura HMAC validada via header `x-elevare-signature`.

**Problema:** a Postman documenta apenas o webhook OUTBOUND (gateway de pagamento → Elevare) em `POST /global-agent/webhook/payment-status` com payload `{quotationId, paymentLinkToken, status, paymentId, paidAmount, currency, timestamp, paymentDetails}` e `status: paid|failed|expired`.

**Perguntas para a Elevare:**

1. Vocês disparam webhooks para nosso endpoint `POST /webhooks/elevare`?
2. Se sim, qual o schema exato do body? Os event types que processamos estao corretos?
3. Qual header vocês usam para assinar? Algoritmo HMAC (sha256)? Qual o formato (hex, base64)?
4. Qual o secret — o mesmo `ELEVARE_WEBHOOK_SECRET` compartilhado, ou algo rotacionado?
5. Como lidar com retries? Vocês reemitem se nao receberem 2xx?

**Acao ao receber resposta:** abrir Story 3.12 para realinhar `elevare-handler.ts`, `elevare-processor.ts`, `elevare-repository.ts` e `tests/unit/elevare-webhook.test.ts` com o contrato oficial.

## 2. Busca de reservas por numero de telefone

**Status no codigo hoje:** `backend/src/integrations/elevare/reservations.ts` usa query param `guestPhone` quando o identifier for um telefone E.164. Ver `detectIdentifierType()` → linha 88.

**Problema:** a Postman do `GET /global-agent/reservations` documenta oficialmente apenas `confirmationNumber`, `hotelId`, `guestName`, `checkInFrom`, `checkInTo`, `limit`, `email`. **`guestPhone` nao aparece.**

**Perguntas para a Elevare:**

1. O endpoint aceita `guestPhone` como filtro? Formato esperado (E.164 com `+`? So digitos?)
2. Se nao aceita, qual a alternativa para buscar reserva a partir de um numero de WhatsApp? Precisamos buscar customer primeiro (`GET /global-agent/customers/phone/{phone}`) e depois usar algum identifier alternativo?

**Acao ao receber resposta:** se `guestPhone` existe → atualizar Postman local e testes. Se nao existe → reescrever `detectIdentifierType` para lookup em duas etapas (customer → reservations).

## 3. Tipo de `offerId`: string vs number

**Status no codigo hoje:**

- `backend/src/integrations/elevare/types.ts:39` declara `offerId: string` em `ElevareOffer` (o que vem do search response)
- `backend/src/integrations/elevare/quotations.ts:11` declara `offerId: number` em `CreateQuotationParams` (o que vai no POST /quotations body)

**Problema:** a Postman mostra `"offerId": 0` no body do POST `/global-agent/quotations` — e valor `number`. Mas nao mostra o tipo que vem na **resposta** do search.

**Perguntas para a Elevare:**

1. O `offerId` na resposta do `/global-agent/search` e string ou number?
2. E no response do `/global-agent/multi-search` (dentro de cada hotel.results)?
3. Se for number, por que `0` e valido (indice?) vs UUID/hash? E indice ordinal dentro do array? E confiavel?

**Acao ao receber resposta:** unificar tipo em `types.ts` e `quotations.ts`. Adicionar coercao se necessario (ex: `Number(offer.offerId)` antes de enviar em quotation).

## 5. Address obrigatorio no customer (validar em teste real)

**Status no codigo hoje:** Story 3.11 implementa payload minimalista `{primaryPhone, firstName, lastName, email?, cpf?, birthDate?}` — **sem `address{}` estruturado**.

**Hipotese a validar:** `address` e coletado no gateway de pagamento (via link), nao no cadastro inicial. O endpoint `/global-agent/customers/phone/{phone}/validate` pode ser apenas um check "esta completo?", nao um gate automatico do `/customers` POST.

**Como validar:** primeiro teste real contra staging/producao com payload minimalista. Se `/customers` POST retornar 400 com mensagem tipo `address required`, **confirmar** que address e obrigatorio.

**Acao:**
- **Se rejeitar:** abrir Story 3.12 com novo payload incluindo `address{street, number, neighborhood, city, state, zipcode, country}`. Atualizar prompts da Stella para coletar endereco antes da cotacao.
- **Se aceitar:** documentar aqui o resultado e seguir com o escopo atual.
