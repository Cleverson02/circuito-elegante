## Descricao

Este story implementa o **buffer de concatenacao de 20 segundos** para mensagens WhatsApp — o componente que resolve o problema das "mensagens picadas" onde hospedes enviam pensamentos fragmentados em rapida sucessao.

**O problema que resolve:**

No WhatsApp, usuarios frequentemente enviam uma idea em 3-5 mensagens consecutivas em poucos segundos (ex: "Oi" + "quero um hotel" + "na serra gaucha" + "para o feriado de junho"). Sem buffer, cada mensagem geraria uma chamada independente ao pipeline de agentes, resultando em 3-5 respostas separadas — uma experiencia terrivel para um concierge de luxo.

**Como funciona:**

1. Webhook handler (Story 4.1) recebe mensagem e chama `buffer.add(phone, parsedMessage)`
2. Buffer acumula mensagens do mesmo usuario (por phone number) em uma janela de 20 segundos
3. Timer resetado a cada nova mensagem do mesmo usuario
4. Apos 20s sem nova mensagem: consolida todas em uma unica string e invoca o pipeline
5. Antes de processar: marca mensagens como "lidas" (FR34) e seta presenca "online" (FR35)
6. Apos pipeline: entrega resposta via Evolution API e seta presenca "offline" (FR35)

**Por que 20 segundos (nao 10, nao 30):**

O valor de 20s vem do PRD (FR12) e foi validado pela Aria na arquitetura. E o sweet spot entre: (a) tempo suficiente para hospede terminar de digitar pensamento completo, (b) nao tao longo que pareca que Stella "sumiu". O timer resetado garante que enquanto o hospede estiver digitando, o buffer espera.

**Decisao arquitetural: In-memory Map + setTimeout.**

Conforme architecture.md secao 9.3, o buffer usa `Map<string, BufferEntry>` em memoria com `setTimeout` para flush. Esta e a escolha correta para MVP (single-instance deployment, NFR8: 5.000 msgs/mes). BullMQ sera usado no Story 4.3 para human-typing simulation, NAO para o buffer. Se escalar para multi-instance, buffer pode migrar para Redis pub/sub — mas isso e FORA do escopo deste story.

**Integracao critica: sessionId resolution.**

O webhook handler (Story 4.1) nao resolve sessionId (campo null nos logs). O buffer e o ponto onde sessionId e resolvido a partir do phone number: `sessionId = phone` (formato E.164 sem +, ex: `5521999999999`). Sessoes existentes sao carregadas do Redis; novas sessoes sao criadas on-demand.

## Acceptance Criteria

- [ ] AC1: Classe `MessageBuffer` implementada em `backend/src/buffer/message-buffer.ts` com metodo `add(phone: string, message: ParsedMessage): void` que acumula mensagens por phone number em janela de 20 segundos
- [ ] AC2: Timer de 20s resetado a cada nova mensagem do mesmo phone number — enquanto hospede digita, buffer espera. Constante `BUFFER_WINDOW_MS = 20_000` exportada e configuravel via env var `MESSAGE_BUFFER_WINDOW_MS` (default 20000)
- [ ] AC3: Apos 20s sem nova mensagem: buffer consolida todas as mensagens em uma unica string (`messages.map(m => m.content).join(' ')`) e invoca callback `onFlush(phone, consolidatedMessage, messageIds)` 
- [ ] AC4: `BufferProcessor` em `backend/src/buffer/buffer-processor.ts` implementa o callback `onFlush` com fluxo: (1) resolver sessionId a partir do phone, (2) construir `remoteJid` = `phone.replace('+', '') + '@s.whatsapp.net'` e marcar mensagens como lidas (FR34 via `markAsReadDeferred(client, remoteJid, messageId, logger)`), (3) setar presenca online (FR35 via `setPresenceOnline`), (4) sanitizar texto consolidado via `sanitizeInput(consolidated)` (anti-injection), (5) chamar `processMessage(PipelineInput)`, (6) entregar resposta via `EvolutionClient.sendText()`, (7) setar presenca offline (FR35 via `setPresenceOffline`)
- [ ] AC5: Session ID resolvido a partir de phone number: `sessionId = phone.replace('+', '')` (ex: `+5521999999999` → `5521999999999`). Se sessao nao existe no Redis, cria nova sessao com `setSession(sessionId, { guestPhone: phone, language: 'pt', hotelId: '', createdAt, updatedAt })`
- [ ] AC6: Webhook handler (`whatsapp-handler.ts`) modificado para chamar `buffer.add(phone, parsedMessage)` apos parse bem-sucedido, em vez de apenas retornar 200 OK. Handler continua retornando 200 OK imediatamente — processamento permanece async
- [ ] AC7: Tratamento de mensagens de media no buffer: mensagens tipo `audio` concatenadas como `[audio]`, mensagens tipo `image` concatenadas como `[imagem: {caption}]` se tiver caption, ou `[imagem]` se nao tiver. Texto concatenado normalmente
- [ ] AC8: Mensagens de um unico texto (sem fragmentacao) tambem passam pelo buffer — aguardam 20s antes de processar. Nao ha bypass para mensagem unica (consistencia de UX)
- [ ] AC9: Log estruturado em dois momentos: (a) `buffer_message_added` quando mensagem entra no buffer (phone, messageId, bufferSize, isNewBuffer), (b) `buffer_flushed` quando buffer consolida (phone, messageCount, totalChars, bufferDurationMs)
- [ ] AC10: Erro no pipeline NAO causa crash — try/catch em `BufferProcessor.onFlush` com log de erro e mensagem fallback ao hospede: "Desculpe, tive uma dificuldade momentanea. Poderia repetir?" via `EvolutionClient.sendText()`
- [ ] AC11: Metodo `buffer.getBufferSize(): number` retorna quantidade de buffers ativos (para health check / monitoring)
- [ ] AC12: Metodo `buffer.flush(phone): void` permite flush manual de um buffer especifico (util para testes e graceful shutdown)
- [ ] AC13: No graceful shutdown (`SIGINT`/`SIGTERM`), todos os buffers ativos sao flushed antes de encerrar — nenhuma mensagem perdida
- [ ] AC14: `backend/src/state/keys.ts` atualizado com chave `buffer` para Redis keys e TTL (para futuro uso, nao usado neste story — buffer e in-memory)
- [ ] AC15: Testes unitarios cobrindo: (a) mensagem unica → flush apos 20s, (b) 3 mensagens em 15s → 1 flush consolidado, (c) timer resetado a cada mensagem, (d) mensagens de phones diferentes nao interferem, (e) media types concatenados corretamente, (f) flush manual, (g) getBufferSize, (h) erro no pipeline → mensagem fallback — cobertura >= 90% branches

### Data Models

- **ParsedMessage** (input do buffer): `{ type: 'text' | 'audio' | 'image', content: string, mediaUrl?: string, phone: string, messageId: string, timestamp: number, isReply: boolean, quotedMessageId?: string }` [Source: backend/src/webhooks/whatsapp-types.ts]
- **BufferEntry** (NOVO, interno): `{ messages: ParsedMessage[], messageIds: string[], timer: NodeJS.Timeout, startedAt: number }` — Map<phone, BufferEntry>
- **PipelineInput** (output do buffer → pipeline): `{ message: string, sessionId: string, sessionContext?: { guestName?, hotelFocus? } }` [Source: backend/src/agents/pipeline.ts:32-39]
- **PipelineResult** (resposta do pipeline): `{ response: string, intent, safetyApproved, multiIntent, latencyMs, hotelFocus? }` [Source: backend/src/agents/pipeline.ts:41-48]
- **SessionData** (sessao Redis): `{ hotelId, guestPhone, language, agentState?, context?, createdAt, updatedAt }` [Source: backend/src/state/session-manager.ts]

### API Specifications

- **Pipeline entry point:** `processMessage(PipelineInput): Promise<PipelineResult>` — chamada sincrona, retorna resposta completa [Source: backend/src/agents/pipeline.ts]
- **Evolution API sendText:** `evolutionClient.sendText(phone, text): Promise<EvolutionResponse>` — envia resposta ao hospede [Source: backend/src/integrations/evolution/client.ts]
- **Session Manager:** `getSession(sessionId)` / `setSession(sessionId, data)` — CRUD Redis com TTL 24h [Source: backend/src/state/session-manager.ts]
- **Sanitize middleware:** `sanitizeInput(message)` em `backend/src/middleware/sanitize.ts` — deve ser aplicado ao texto consolidado antes de enviar ao pipeline (max 2000 chars, anti-injection) [Source: architecture.md#section-10]

### Technical Constraints

- **In-memory buffer** — nao persiste entre restarts. Graceful shutdown flush mitiga, mas hard crash perde mensagens em buffer. Aceitavel para MVP (NFR8: 5.000 msgs/mes) [Source: architecture.md#section-9.3]
- **Single-instance only** — buffer in-memory nao funciona com multiple instances. Para escalar: migrar para Redis XADD/XREAD ou BullMQ delayed jobs. FORA do escopo [Source: architecture.md#section-12 — single droplet]
- **Sanitizacao do texto consolidado** — aplicar `sanitizeInput()` ao texto consolidado ANTES de enviar ao pipeline. Mensagens individuais ja passaram pelo webhook, mas a concatenacao pode criar novos patterns de injection [Source: architecture.md#section-10]
- **NFR7 compliance** — O buffer adiciona 20s de latencia proposital. Apos consolidacao, o pipeline deve responder em <= 5s (NFR1). Total: ~25s max perceived latency para primeira mensagem [Source: PRD NFR7]

### Integracao com Stories Subsequentes

- **Story 4.3 (Typing Simulation):** Substituira o `evolutionClient.sendText()` direto no BufferProcessor por enfileiramento no BullMQ typing queue. Interface do processor deve ser facil de trocar (dependency injection)
- **Story 4.4 (Renderizacao Premium):** Quando pipeline retornar opcoes com imagens, a entrega sera via `sendMedia()` em vez de `sendText()`. BufferProcessor precisara ser extendido
- **Story 4.6 (Chat Widget):** Website NAO usa buffer 20s — processamento imediato. O chat endpoint existente (`/api/v1/chat`) ja faz isso
