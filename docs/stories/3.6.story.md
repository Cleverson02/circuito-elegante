## Descrição

Este story implementa o motor de **Upsell Elegante** — a capacidade de Stella de identificar, silenciosamente, quando uma opção premium está ao alcance do hóspede e sinalizar essa oportunidade ao Persona Agent para que ele a apresente com a sutileza que o luxo exige.

**Filosofia:** O upsell em hotelaria de alto padrão não é venda — é curadoria. Quando um hóspede busca um quarto Standard e existe uma Suite com vista para o mar por apenas R$ 80 a mais por noite, omitir essa informação seria um desserviço. O que Stella faz é exatamente o que um concierge humano excepcional faria: mencionar a possibilidade com naturalidade, sem insistência, sem pressão, sem artificialidade. "A propósito, por apenas R$ 80 a mais por noite, o senhor poderia desfrutar de uma suíte com vista para o oceano..." — e se o hóspede ignora ou declina, o assunto morre ali. Para sempre naquela sessão.

**Problema/Necessidade:** A curadoria (Story 3.4) já ranqueia e filtra ofertas. Porém, o sistema atualmente apresenta apenas as opções curadas sem destacar oportunidades de upgrade próximas ao preço da escolha provável. Sem o upsell elegante, Stella perde oportunidades legítimas de agregar valor ao hóspede enquanto incrementa a receita do hotel — o verdadeiro win-win da hotelaria de luxo.

**Decisão Arquitetural:** O motor de upsell é uma **função pura** — recebe a lista curada de ofertas e retorna um `UpsellHint` sem side effects. A função NÃO gera mensagens; ela produz dados estruturados que o Persona Agent consome para compor prosa apropriada ao idioma e tom. O controle de "one-shot" (não repetir sugestão após recusa/ignorar) é gerenciado via Redis com chave de sessão. O threshold de preço é configurável (default: 15%) para permitir ajuste fino sem deploy.

**Fluxo:**
```
Curadoria (Story 3.4) → Lista curada + ranking
  → UpsellEngine.detect(offers, guestSelection)
  → UpsellHint { detected, original, upgrade, priceDiff, reason }
  → Redis check: upsell_offered:{sessionId} exists?
    → YES: suppress hint (one-shot rule)
    → NO: pass hint to Persona Agent
  → Persona Agent crafts luxury prose (language-aware)
  → Redis SET upsell_offered:{sessionId} = true (TTL = session TTL)
```

## Dev Notes

### Upsell Detection Algorithm — Deep Dive

**Input:** Array de `CuratedOffer[]` (output da Curadoria, Story 3.4) + `selectedIndex` (qual oferta o hóspede indicou preferência).

**Threshold:** Configurável, default 15%. A fórmula:
```
priceDiffPercent = ((upgrade.price - original.price) / original.price) * 100
isQualified = priceDiffPercent <= UPSELL_THRESHOLD_PERCENT
```

**Seleção do melhor upgrade:** Quando múltiplas ofertas qualificam (dentro do threshold), o motor seleciona a de **MENOR** `priceDiff` — a filosofia é custo-benefício para o hóspede, não revenue maximization para o hotel. Isso é intencional e alinhado com o PRD §3.3 (Gentle Upsell).

```typescript
// Pseudocode — implementação real em upsell.ts
function detectUpsell(
  offers: CuratedOffer[],
  selectedIndex: number,
  config: UpsellConfig = getUpsellConfig()
): UpsellHint {
  // Guard clauses
  if (offers.length <= 1) return NO_UPSELL;
  if (selectedIndex < 0 || selectedIndex >= offers.length) return NO_UPSELL;

  const selected = offers[selectedIndex];

  // All same price?
  const allSamePrice = offers.every(o => o.price === selected.price);
  if (allSamePrice) return NO_UPSELL;

  // Already premium?
  const maxPrice = Math.max(...offers.map(o => o.price));
  if (selected.price === maxPrice) return NO_UPSELL;

  // Find qualified upgrades
  const candidates = offers
    .filter(o => o.price > selected.price)
    .map(o => ({
      ...o,
      priceDiff: o.price - selected.price,
      priceDiffPercent: ((o.price - selected.price) / selected.price) * 100,
    }))
    .filter(o => o.priceDiffPercent <= config.thresholdPercent)
    .sort((a, b) => a.priceDiff - b.priceDiff); // Ascending — smallest diff first

  if (candidates.length === 0) return NO_UPSELL;

  const best = candidates[0];
  return {
    detected: true,
    original: { offerId: selected.offerId, price: selected.price, name: selected.name },
    upgrade: {
      offerId: best.offerId,
      price: best.price,
      name: best.name,
      priceDiff: best.priceDiff,
      priceDiffPercent: Math.round(best.priceDiffPercent * 100) / 100, // 2 decimal places
    },
    reason: deriveReason(selected, best),
  };
}
```

### Threshold Configuration

```typescript
// backend/src/config/upsell.ts
const UPSELL_THRESHOLD_PERCENT = (() => {
  const envVal = process.env.UPSELL_THRESHOLD_PERCENT;
  const parsed = envVal ? Number(envVal) : 15;
  if (isNaN(parsed) || parsed < 1 || parsed > 50) {
    throw new ConfigValidationError(
      `UPSELL_THRESHOLD_PERCENT must be between 1 and 50, got: ${envVal}`
    );
  }
  return parsed;
})();
```

### Redis Key for One-Shot Enforcement

```
upsell_offered:{sessionId}  →  "true"  (TTL: 24h = session TTL)
```

**Lifecycle:**
1. Curadoria retorna ofertas → `detectUpsell()` analisa → hint produzido
2. Orchestrator calls `shouldOfferUpsell(sessionId)` → checks Redis
3. If `false` (not yet offered): pass hint to Persona Agent
4. Persona Agent crafts prose, sends to guest
5. `markUpsellOffered(sessionId)` → SET in Redis
6. Any subsequent call to `shouldOfferUpsell()` returns `false` → hint suppressed
7. Key expires with session (24h TTL)

**Important:** `markUpsellOffered()` is called AFTER the message is sent, not after detection. If the Persona Agent decides not to include the upsell in the message (for any reason), the one-shot is NOT consumed.

### Persona Agent Integration — What Data to Pass

The `UpsellHint` is passed to the Persona Agent as part of the tool response data. The Persona Agent's system prompt already knows how to handle upsell hints:

```typescript
// What the Persona Agent receives (example):
{
  detected: true,
  original: { offerId: 42, price: 850, name: "Quarto Superior" },
  upgrade: {
    offerId: 67,
    price: 930,
    name: "Suíte Vista Mar",
    priceDiff: 80,
    priceDiffPercent: 9.41,
  },
  reason: "ocean-view suite with balcony"
}
```

The Persona Agent then crafts language-appropriate prose:
- **PT:** "A propósito, por apenas R$ 80 a mais por noite, o senhor poderia desfrutar de uma suíte com vista para o mar e varanda privativa..."
- **EN:** "By the way, for just $80 more per night, you could enjoy an ocean-view suite with a private balcony..."
- **ES:** "Por cierto, por solo $80 más por noche, podría disfrutar de una suite con vista al mar y balcón privado..."

The upsell engine does NOT know about languages. It provides raw data. Persona Agent handles all i18n.

### Anti-Patterns to AVOID

| Anti-Pattern | Why It's Wrong | What to Do Instead |
|-------------|----------------|-------------------|
| `availableRooms: 2` in hint | Creates artificial scarcity pressure (PRD §3.2: NEVER) | Omit availability data entirely |
| `"Apenas 2 quartos!"` in reason | Scarcity trigger violates PRD §3.3 | `reason` is structured data, not prose |
| `urgency: "high"` field | Urgency pressure violates tone rules | No urgency field in type |
| Retry after guest ignores | Insistence violates one-shot rule | `shouldOfferUpsell()` blocks repeat |
| Select most expensive upgrade | Revenue-first contradicts luxury philosophy | Select SMALLEST price difference |
| Format price in hint | Currency formatting is Persona Agent's job | Return raw number |
| Generate message in engine | Violates separation of concerns | Return `UpsellHint` data only |

### Session-Scoped One-Shot Logic — Edge Cases

| Scenario | Behavior |
|----------|----------|
| Guest ignores upsell mention | `markUpsellOffered()` was called → no repeat in this session |
| Guest explicitly declines ("no thanks") | Already marked → no repeat |
| Guest asks about the upgrade | This is NOT a new upsell attempt — it's a follow-up conversation about the offered option. Allowed |
| New session (next day) | New `sessionId` → fresh start, upsell can be offered again |
| Guest searches for different hotel in same session | One-shot is per SESSION, not per search. Intentional: avoid upsell fatigue |
| Server restart mid-session | Redis persists → one-shot state preserved |

### `deriveReason()` Helper

The `reason` field should describe what makes the upgrade special compared to the original selection:

```typescript
function deriveReason(original: CuratedOffer, upgrade: CuratedOffer): string {
  // Use category and amenities to describe the upgrade value
  // This is structured data, not prose
  const parts: string[] = [];
  if (upgrade.category && upgrade.category !== original.category) {
    parts.push(upgrade.category);
  }
  if (upgrade.amenities) {
    const originalAmenities = new Set(original.amenities ?? []);
    const newAmenities = upgrade.amenities.filter(a => !originalAmenities.has(a));
    if (newAmenities.length > 0) {
      parts.push(...newAmenities);
    }
  }
  return parts.length > 0 ? parts.join(', ') : upgrade.name;
}
```
