## Descricao

Este story implementa os **testes end-to-end multi-canal** — a validacao final do EPIC-4 que garante que todos os fluxos funcionam de ponta a ponta antes de seguir para o EPIC-5.

**Objetivo:** Validar que o sistema Stella processa corretamente mensagens via WhatsApp e Website, incluindo buffer, typing simulation, renderizacao premium, correfencia via Reply, business hours, e template messages. E o quality gate do EPIC-4.

**Estrategia de testes:** Testes de integracao com mocks das APIs externas (Evolution API, Elevare API) mas com os componentes internos reais (buffer, pipeline, queue, worker, WebSocket). Usa supertest + ws para HTTP/WebSocket, BullMQ em modo teste.

[Source: docs/stories/epics/EPIC-4-EXECUTION.yaml#4.8]

## Acceptance Criteria

- [x] AC1: **Fluxo WhatsApp E2E** — Simular: webhook recebe mensagem → buffer 20s → pipeline processa → typing queue → worker envia resposta. Verificar que `evolutionClient.sendText()` e chamado com texto correto e delays apropriados.
- [x] AC2: **Buffer 20s** — Enviar 3 mensagens em 15s via webhook simulado → verificar que buffer consolida em 1 mensagem → pipeline recebe texto concatenado.
- [x] AC3: **Typing Simulation** — Resposta longa (>300 chars) → verificar que e quebrada em chunks → delays calculados corretamente → chunks enviados em ordem com composing events.
- [x] AC4: **Renderizacao Premium** — Pipeline retorna CuratedOptions → verificar que `renderCuratedOptions()` gera chunks de imagem + legenda para WhatsApp, e chunks de texto para website.
- [x] AC5: **Correfencia Reply** — Enviar opcoes curadas → capturar messageIds → simular Reply a imagem 2 → verificar resolucao para offerId correto via `resolveCoreference()`.
- [x] AC6: **Website Widget E2E** — Conectar WebSocket a `/ws/chat` → enviar mensagem → receber resposta chunked via WebSocket → verificar formato `{ type: 'message', text, chunkIndex, totalChunks }`.
- [x] AC7: **Fora de Expediente** — Mock de hora 23:00 BRT → enviar mensagem WhatsApp → verificar que recebe fallback message → lead registrado em storage.
- [x] AC8: **Template Messages** — Chamar `sendTemplateMessage('follow_up_quote_expiring', variables)` → verificar payload correto enviado ao Evolution API → verificar fallback quando template nao aprovado (erro 470).
- [x] AC9: **Graceful Shutdown** — Iniciar shutdown → verificar que buffers sao flushed, WebSocket connections fechadas com code 1001, queues encerradas.
- [x] AC10: **Todos os testes passam** — Suite completa verde, sem flaky tests. Cada teste isolado (setup/teardown limpa estado).
- [x] AC11: Testes organizados em `tests/e2e/` com estrutura: `whatsapp-flow.e2e.test.ts`, `website-flow.e2e.test.ts`, `out-of-hours.e2e.test.ts`, `template-messages.e2e.test.ts`.

## Dev Notes

### Testing Strategy
- **Mocks:** Evolution API e Elevare API mockados via jest.fn(). Redis e BullMQ reais em modo teste.
- **Isolation:** Cada teste usa sessionId unico, cleanup no afterEach.
- **Timeouts:** Testes de buffer precisam de timeout > 20s (buffer window). Jest timeout configurado para 30s nos testes E2E.
- **WebSocket client:** Usar pacote `ws` diretamente para conectar ao server em teste.

- 4.5 (Reply), 4.6 (Widget), 4.7 (Out-of-Hours), 4.9 (Templates)
