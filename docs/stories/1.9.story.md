## Descrição

Refinar o system prompt, a lógica de orchestration, e as técnicas de concierge da Stella para transformá-la na melhor concierge digital do mundo para o segmento de hotéis de luxo no Brasil. Implementar hierarquia de fontes de dados, controle anti-alucinação rigoroso, técnicas profissionais de concierge (proatividade, cross-selling, personalização), e resposta adaptada por perfil de hóspede.

**Problema/Necessidade:** Com as Stories 1.7 e 1.8, a Stella terá acesso a todos os dados (estruturados e semânticos). Porém, ter acesso a dados não é o mesmo que saber usá-los como uma concierge de classe mundial. A Stella precisa de técnicas profissionais de atendimento, controle rigoroso para nunca inventar informação, e inteligência para combinar múltiplas fontes de dados na resposta ideal.

**Visão:** A Stella deve ser indistinguível de um concierge humano de hotel 5 estrelas — proativa, conhecedora, calorosa, precisa, e absolutamente confiável. Quando não tem a informação, admite elegantemente e oferece verificar.

## Acceptance Criteria

### Hierarquia de Fontes (Anti-Alucinação)

- [ ] AC1: Stella segue hierarquia rígida de fontes ao responder:
  1. **JSONB estruturado** (tool `query_hotel_details`) — dados verificados, prioridade máxima
  2. **RAG semântico** (tool `query_knowledge_base`) — dados vetorizados, alta confiança (similarity > 0.78)
  3. **Resposta genérica contextualizada** — baseada no tipo/categoria do hotel, sem dados específicos inventados
  4. **Admissão elegante + oferta de verificação** — "Vou confirmar essa informação diretamente com o {hotel} e retorno para você"
  5. **Handover humano** — quando a questão é complexa demais ou envolve exceções
- [ ] AC2: Stella NUNCA inventa dados específicos (preços, horários, nomes de restaurantes, políticas). Se o dado não existe nas fontes, ela NÃO fabrica.
- [ ] AC3: Quando RAG retorna similarity < 0.78, Stella não usa o resultado como facto — trata como "informação possível" e oferece verificar

### Técnicas de Concierge Profissional

- [ ] AC4: System prompt inclui técnicas de concierge de hotel 5 estrelas:
  - **Proatividade:** Antecipar necessidades ("Como vocês vão com crianças, vale mencionar que o {hotel} tem programação infantil especial")
  - **Cross-selling contextual:** Sugerir experiências complementares baseadas no perfil ("Já que vocês gostam de gastronomia, o {hotel} oferece harmonização de vinhos")
  - **Personalização por perfil:** Adaptar linguagem e sugestões por tipo de hóspede (casal romântico, família com crianças, corporativo, aventureiro)
  - **Rapport:** Usar nome do hóspede quando disponível, retomar contexto de conversas anteriores
  - **Urgência elegante:** Informar disponibilidade limitada sem pressão ("Esse período costuma ter alta procura, sugiro confirmarmos logo")
- [ ] AC5: Stella usa os `sales_arguments_by_profile` do JSONB para personalizar argumentos por tipo de hóspede
- [ ] AC6: Stella usa `objections` e `attention_points` do JSONB para antecipar e responder objeções comuns

### Orchestration Inteligente

- [ ] AC7: Intent Agent atualizado para classificar com granularidade fina:
  - `DETAIL_QUERY` → `query_hotel_details` (dados estruturados)
  - `OPEN_QUESTION` → `query_knowledge_base` (RAG semântico)
  - `SEARCH` → `search_hotels` (filtros)
  - `COMPARISON` → `query_hotel_details` × N hotéis (comparação lado a lado)
  - `BOOKING` → Elevare API ou handover
  - `CHAT` → Resposta direta com persona
  - `HANDOVER` → Transferência humana
- [ ] AC8: Orchestrator combina múltiplas tools quando necessário:
  - "Me conta sobre o Fasano" → `query_hotel_details` (estruturado) + `query_knowledge_base` (FAQ/experiências)
  - "Compara Fasano com Emiliano" → 2x `query_hotel_details` + formatação comparativa
- [ ] AC9: Resposta da Stella é formatada para WhatsApp (sem Markdown pesado, emojis moderados, parágrafos curtos, máximo 400 chars por mensagem antes de chunk)

### Qualidade da Resposta

- [ ] AC10: Respostas incluem source attribution interna (log): qual tool forneceu cada dado
- [ ] AC11: Fallback chain testado: quando JSONB não tem dado → tenta RAG → resposta genérica → oferta de verificação
- [ ] AC12: Testes de regressão: 20 perguntas-tipo cobrindo todas as categorias da taxonomia, verificando que respostas são factuais e não alucinadas

## Decisões Técnicas (@architect)

### Threshold de Similarity
- **0.78** (não 0.7 como estava) — baseado em benchmarks de RAG para domínio hoteleiro em português
- Abaixo de 0.78: Stella trata como "informação possível" e confirma com linguagem hedge
- Abaixo de 0.6: Stella ignora o resultado completamente

### Formatação WhatsApp
- Máximo 400 chars por chunk de mensagem (BullMQ Human-Typing Queue já faz o chunking)
- Emojis: uso moderado e contextual (não em cada frase)
- Sem Markdown heavy: negrito com * é ok, mas sem headers, tables, code blocks
- Parágrafos curtos (2-3 frases máx)

### Anti-Alucinação: Abordagem por Design
- Não depender apenas do prompt ("não invente") — isso falha
- Implementar na lógica: se a tool retornou vazio/low-confidence, o código força o fallback
- Log de source attribution permite auditoria posterior

### Perfis de Hóspede
Usar os 4 perfis padrão que já existem nos JSONs de enrichment:
1. **Casal romântico** — foco em privacidade, experiências a dois, spa
2. **Família com crianças** — foco em programação infantil, segurança, espaço
3. **Corporativo** — foco em eficiência, Wi-Fi, salas de reunião
4. **Aventureiro** — foco em atividades, natureza, experiências únicas

Os 7 intents atuais são **expandidos** (não substituídos) para maior granularidade:

| `CHAT` | `CHAT` | Mantido |
| `MULTIMODAL` | `MULTIMODAL` | Mantido |

| — | `DETAIL_QUERY` | **NOVO** — query_hotel_details |
| — | `COMPARISON` | **NOVO** — query_hotel_details × N |

O Intent Agent deve manter backwards compatibility: se receber um intent antigo via log/teste, deve mapear para o novo automaticamente.

### Suite de Teste de Regressão

As 20 perguntas-tipo devem ser implementadas como **script automatizado** (`backend/src/tests/regression/concierge-regression.test.ts`):
- Executa cada query contra o pipeline completo (Intent → Orchestrator → Tools → Persona)
- Valida: (1) resposta não contém dados fabricados, (2) tool correta foi chamada, (3) source attribution presente nos logs
- Pode rodar como parte do CI com `npm run test:regression`
